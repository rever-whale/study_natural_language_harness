# 13. Claude, Codex 로컬 세션 운영법

이 책은 로컬에서 Claude나 Codex 같은 에이전트 도구를 사용하는 상황을 전제로 한다. 중요한 것은 특정 도구의 기능보다 세션 운영 방식이다. 도구가 달라도 자연어 하네스 문서는 그대로 쓸 수 있어야 한다.

## 세션 시작 프롬프트

```text
이 프로젝트에서는 ai-harness/를 팀 공용 하네스 지식베이스로 사용한다.

먼저 다음 문서를 읽어줘:
- ai-harness/README.md
- ai-harness/PROJECT_CONTEXT.md
- ai-harness/AI_WORKFLOW.md
- ai-harness/EVAL_RUBRIC.md

아직 작업하지 말고, 이 프로젝트에서 AI 세션을 진행할 때 지켜야 할 규칙만 요약해줘.
```

세션을 시작할 때 작업부터 던지지 않는다. 먼저 환경을 불러온다.

## 도구별 운영 차이

자연어 하네스는 도구 중립적이어야 하지만, 세션 운영에서는 차이를 고려한다.

| 도구 유형 | 강점 | 주의점 | 하네스 대응 |
| --- | --- | --- | --- |
| Claude Code 계열 | 긴 대화와 파일 작업에 강함 | 세션이 길어지며 초기 제약이 흐려질 수 있음 | 중간 압축과 세션 로그를 강제 |
| Codex 계열 | 코드베이스 탐색, 패치, 검증 루프에 강함 | 명령 실행과 파일 변경 권한 관리가 중요 | 권한 규칙과 검증 기준을 명확히 제공 |
| IDE Chat 계열 | 짧은 수정과 설명에 빠름 | 프로젝트 전체 맥락 유지가 약할 수 있음 | 작은 컨텍스트 패킷과 작업 헌장 사용 |
| 웹 챗 계열 | 문서 초안, 사고 정리에 편함 | 로컬 파일과 실제 검증 연결이 약함 | 결과를 세션 로그와 검증 요청으로 분리 |

팀은 도구별 프롬프트를 완전히 따로 만들 필요는 없다. 다만 `AI_WORKFLOW.md` 안에 “이 도구를 쓸 때 특히 조심할 점”을 둔다.

## 작업 투입 프롬프트

```text
이번 작업의 작업 헌장은 다음 파일이다:
- ai-harness/tasks/2026-06-auth-email-verification.md

진행 방식:
1. 작업 헌장을 읽고 완료 조건을 요약한다.
2. 컨텍스트 패킷에 포함된 문서를 읽는다.
3. 모호한 점이 있으면 질문한다.
4. 질문이 없으면 5단계 이내 계획을 제시한다.
5. 수행 후 평가 루브릭으로 자체 점검한다.
6. 마지막에 SESSION_LOG 초안을 작성한다.
```

이 프롬프트는 도구 중립적이다. Claude에서도 Codex에서도 쓸 수 있다.

## 세션 모드

모든 세션을 같은 방식으로 운영하면 비효율적이다. 작업 유형에 따라 모드를 선언한다.

```text
이번 세션 모드: Read-only Review

규칙:
- 파일을 수정하지 않는다.
- 먼저 관련 문서를 읽고 위험과 누락을 찾는다.
- 최종 응답은 Findings, Questions, Suggested Changes로 나눈다.
```

```text
이번 세션 모드: Assisted Implementation

규칙:
- 작업 헌장과 컨텍스트 패킷을 먼저 읽는다.
- 변경 계획을 제시한 뒤 진행한다.
- 변경 후 완료 조건별 검증 결과를 남긴다.
```

```text
이번 세션 모드: Harness Maintenance

규칙:
- 제품 코드는 건드리지 않는다.
- 세션 로그와 회귀 노트를 검토한다.
- 하네스 문서 변경 제안만 작성한다.
```

모드를 선언하면 에이전트가 “무엇을 해도 되는지”를 더 잘 이해한다.

## 장기 세션에서의 압축 규칙

세션이 길어지면 중요한 맥락이 희미해진다. 중간에 요약을 요청한다.

```text
지금까지의 세션을 다음 형식으로 압축해줘.

- 원래 목표:
- 확정된 결정:
- 변경된 범위:
- 이미 검증한 것:
- 아직 검증하지 못한 것:
- 다음 작업자가 반드시 알아야 할 것:
- 하네스 문서에 반영할 것:
```

이 요약은 다음 세션의 입력이 된다. 압축을 모델의 자동 기능에만 맡기지 말고, 팀이 필요한 항목을 지정한다.

## 권한과 위험

로컬 에이전트가 파일을 수정하거나 명령을 실행할 수 있다면, 자연어 하네스에 권한 규칙을 둔다.

```text
권한 규칙:

- 사용자 데이터 삭제, 마이그레이션, 배포 명령은 승인 전 실행 금지.
- 보안 설정, 인증 정책, 결제 로직 변경은 먼저 계획과 위험을 보고.
- 대량 파일 변경은 변경 대상 목록을 먼저 제시.
- 외부 네트워크 접근이 필요하면 목적과 대상 URL을 설명.
```

권한은 “조심해줘”가 아니라 구체적인 승인 조건으로 써야 한다.

## 세션 종료 전 인수인계

로컬 에이전트 세션은 중간에 끊기거나 다른 팀원이 이어받을 수 있다. 그래서 종료 전에는 반드시 인수인계를 남긴다.

```text
다음 사람이 이어받을 수 있도록 인수인계를 작성해줘.

포함할 것:
- 현재 목표
- 이미 확정된 사실
- 내가 한 변경 또는 제안
- 검증한 항목
- 검증하지 못한 항목
- 위험하거나 되돌릴 수 없는 작업
- 다음 사람이 처음 읽어야 할 문서
```

이 인수인계는 단순 요약보다 강하다. 다음 세션의 컨텍스트 패킷으로 바로 들어갈 수 있기 때문이다.
