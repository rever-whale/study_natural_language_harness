# 4. 실패하는 프롬프트의 공통 구조

프롬프트 실패는 모델의 지능 부족만으로 설명되지 않는다. 실무에서 반복되는 실패는 대부분 입력 계약이 약할 때 생긴다. 이 장에서는 팀이 지식베이스에 기록해야 할 실패 모드를 정리한다.

## 목표가 결과물이 아니라 활동으로 쓰인 경우

나쁜 목표는 “리팩터링해줘”, “문서화해줘”, “개선해줘”처럼 활동만 말한다. 좋은 목표는 완료 상태를 말한다.

```text
나쁜 예:
이 컴포넌트를 개선해줘.

좋은 예:
이 컴포넌트에서 상태 계산과 렌더링 책임을 분리하고,
기존 테스트가 통과하며,
사용자에게 보이는 UI 텍스트와 동작은 바뀌지 않게 해줘.
```

목표가 활동이면 모델은 “무언가 했다”로 끝낸다. 목표가 상태이면 검증할 수 있다.

## 제약이 암묵지로 남은 경우

사람은 팀의 암묵적 규칙을 안다고 착각한다. 모델은 모른다. 예를 들어 “우리 프로젝트에서는 모달 안에 또 카드 UI를 넣지 않는다”, “이 API는 하위 호환 때문에 필드명을 바꾸면 안 된다” 같은 규칙은 매번 주입되거나 프로젝트 컨텍스트에 있어야 한다.

자연어 하네스의 첫 번째 임무는 암묵지를 명시지로 바꾸는 것이다.

## 입력 자료가 너무 많거나 너무 적은 경우

너무 적으면 모델은 빈칸을 추측한다. 너무 많으면 중요한 제약이 묻힌다. 컨텍스트 패킷은 “가능한 모든 자료”가 아니라 “이 작업에 필요한 최소 충분 자료”여야 한다.

작업 시작 전에 다음을 분리한다.

- 반드시 읽을 자료
- 필요하면 읽을 자료
- 읽지 말아야 할 오래된 자료
- 현재 믿을 수 없는 자료

특히 오래된 설계 문서는 위험하다. 지식베이스에는 문서의 최신성도 함께 기록해야 한다.

## 검증을 모델의 자기평가에 맡긴 경우

“잘 됐는지 확인해줘”는 약한 검증이다. 모델은 자기가 만든 결과를 낙관적으로 평가할 수 있다. 검증은 구체적이어야 한다.

```text
검증 기준:
- 기존 사용자 흐름이 깨지지 않았는가?
- API 응답 필드가 바뀌지 않았는가?
- 접근성 라벨이 유지되는가?
- 에러 케이스가 문서화되었는가?
- 변경하지 않기로 한 파일을 건드리지 않았는가?
```

자연어 하네스는 검증 질문을 미리 준비해 둔다. 평가 기준이 있어야 결과물이 팀 기준에 닿는다.

## 세션의 배움이 저장되지 않는 경우

AI 세션에서 가장 아까운 손실은 결과물이 아니라 배움의 손실이다. 왜 실패했는지, 어떤 제약을 추가해야 하는지, 어떤 프롬프트가 효과 있었는지가 저장되지 않으면 다음 사람도 같은 비용을 낸다.

따라서 모든 중요한 세션은 마지막에 다음 세 줄을 남긴다.

```text
이번 세션에서 새로 알게 된 프로젝트 사실:
다음 세션부터 컨텍스트에 추가해야 할 규칙:
하네스 문서에서 수정할 항목:
```

이 세 줄이 자연어 하네스를 성장시킨다.
