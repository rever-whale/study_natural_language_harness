# 10. 평가 루브릭과 회귀 체크

AI 결과물을 신뢰하려면 평가 기준이 필요하다. 자연어 하네스에서 평가 루브릭은 자동 채점기가 아니라 팀이 결과물을 보는 공통 렌즈다.

## 평가 루브릭 기본형

```text
# EVAL_RUBRIC

## 공통 기준

- 목표 적합성: 작업 헌장의 목표를 충족하는가?
- 범위 준수: 비목표와 금지 범위를 침범하지 않았는가?
- 프로젝트 일관성: 기존 아키텍처, 용어, 문체와 맞는가?
- 검증 가능성: 결과가 증거와 함께 제시되는가?
- 운영 위험: 배포, 보안, 개인정보, 장애 대응 관점의 위험이 관리되는가?

## 판정

- Pass: 바로 수용 가능
- Needs Review: 사람이 특정 항목을 확인해야 함
- Revise: 재작업 필요
- Reject: 방향이 틀렸거나 위험함
```

판정 단계가 있어야 AI 결과가 “괜찮아 보임”에서 멈추지 않는다.

## 작업 유형별 루브릭

공통 루브릭만으로는 부족하다. 작업 유형별로 기준을 추가한다.

- 기능 설계: 사용자 흐름, 예외 케이스, 데이터 계약, 마이그레이션 위험
- 버그 수정: 재현 조건, 원인과 증상의 구분, 회귀 방지
- 문서 작성: 독자, 최신성, 근거, 용어 일관성
- 코드 리뷰: 정확성, 변경 범위, 테스트, 보안, 유지보수성
- 장애 분석: 타임라인, 영향 범위, 근본 원인, 후속 조치

팀은 자주 하는 작업부터 루브릭을 만든다. 모든 것을 한 번에 만들 필요는 없다.

## 작업 유형별 상세 루브릭 예시

다음 예시는 그대로 복사해 팀에 맞게 줄여 쓸 수 있다.

```text
## 버그 수정 루브릭

- 재현성: 버그가 발생하는 조건이 분명히 설명되었는가?
- 원인 구분: 근본 원인과 관찰된 증상을 구분했는가?
- 범위 제한: 수정이 문제 영역 밖으로 번지지 않았는가?
- 회귀 방지: 과거 유사 실패 항목을 확인했는가?
- 사용자 영향: 수정 전후 사용자 경험이 설명되었는가?
- 검증 증거: 재현 케이스, 테스트, 수동 확인 중 하나 이상의 증거가 있는가?

판정:
- Pass: 원인, 수정, 검증 증거가 모두 있음
- Needs Review: 검증은 있으나 원인 설명이 불완전함
- Revise: 수정은 했지만 재현/검증이 없음
- Reject: 증상만 가렸거나 범위를 과도하게 넓힘
```

```text
## 기능 명세 루브릭

- 사용자 정의: 주요 사용자와 제외 사용자가 구분되었는가?
- 완료 조건: "무엇을 만든다"가 아니라 "어떤 상태가 된다"로 쓰였는가?
- 비목표: 이번 범위에서 하지 않을 일이 명확한가?
- 데이터 계약: API, 이벤트, 권한, 저장 데이터의 변화가 정리되었는가?
- 예외 흐름: 실패, 권한 없음, 빈 상태, 지연 상태가 포함되었는가?
- 검증 계획: 출시 전 확인할 항목이 역할별로 나뉘었는가?
```

```text
## 코드 리뷰 루브릭

- 의도 일치: 변경이 PR 설명의 목적과 맞는가?
- 변경 범위: 필요 이상으로 넓은 파일 변경이 없는가?
- 기존 패턴: 프로젝트의 컴포넌트, API, 테스트 스타일을 따르는가?
- 위험 구역: 인증, 결제, 권한, 개인정보, 배포 설정 변경이 식별되었는가?
- 테스트 적합성: 변경 위험에 비례하는 테스트 또는 검증이 있는가?
- 리뷰 가능성: 사람이 확인해야 할 판단과 기계적으로 확인 가능한 항목이 분리되었는가?
```

루브릭은 완벽한 평가자가 아니다. 하지만 AI가 결과물을 낼 때 무엇을 증명해야 하는지 알려주는 계약이다.

## 회귀 체크

회귀 체크는 과거에 실패했던 항목을 다시 확인하는 절차다.

```text
# REGRESSION_NOTES

## 인증 플로우

- 2026-04: 에러 문구를 바꿔 고객지원 매크로가 깨짐.
  - 체크: 인증 관련 문구 변경 시 support-macro 문서 확인.

- 2026-05: 게스트 사용자의 리다이렉트 루프가 재발함.
  - 체크: 게스트, 로그인 사용자, 만료 세션을 분리해서 검증.
```

회귀 노트는 팀이 실제로 아팠던 지점이다. 일반적인 베스트 프랙티스보다 우선순위가 높다.

## 회귀 노트의 승격 절차

모든 실패를 영구 규칙으로 만들면 하네스가 무거워진다. 실패 기록은 세 단계로 관리한다.

1. `SESSION_LOG`에 실패를 기록한다.
2. 같은 유형의 실패가 반복되거나 영향이 컸다면 `REGRESSION_NOTES.md`로 승격한다.
3. 회귀 체크가 세 달 동안 더 이상 의미 없거나 시스템이 바뀌었다면 `deprecated`로 표시한다.

회귀 노트에는 감상이 아니라 확인 행동이 있어야 한다.

```text
나쁜 기록:
- AI가 인증을 자꾸 헷갈림.

좋은 기록:
- 만료 세션과 권한 없음 상태를 혼동한 수정이 2회 발생.
  - 체크: 인증 관련 작업은 `expired session`, `unauthorized`, `forbidden`을 별도 케이스로 검증한다.
  - 근거: sessions/2026-05-12-auth-loop.md, sessions/2026-05-28-role-guard.md
```

## AI에게 평가를 맡길 때의 주의

AI에게 자기 결과를 평가시킬 수는 있지만, 판정은 조심해야 한다. 좋은 프롬프트는 AI가 “통과”를 쉽게 말하지 못하게 한다.

```text
아래 결과를 평가 루브릭으로 검토해줘.
증거가 없는 항목은 통과로 판정하지 말고 "미확인"으로 표시해줘.
수정이 필요한 항목은 중요도와 함께 정리해줘.
```

자연어 하네스의 목적은 AI를 불신하는 것이 아니다. 신뢰할 수 있는 방식으로 의심하는 것이다.

## 평가 결과 저장 형식

평가 결과는 다음 형식으로 저장하면 나중에 하네스 개선에 재사용하기 좋다.

```text
# 평가 결과

작업:
평가일:
평가자:

## 판정

Overall: Needs Review

## 완료 조건별 결과

| 완료 조건 | 판정 | 증거 | 비고 |
| --- | --- | --- | --- |
| 게스트 접근 차단 | Pass | 수동 확인 로그 |  |
| 로그인 사용자 세션 유지 | Unverified | 증거 없음 | 추가 확인 필요 |
| 에러 문구 유지 | Pass | diff 확인 |  |

## 회귀 체크

- 인증 에러 문구: Pass
- 게스트 리다이렉트 루프: Needs Review

## 다음 조치

- 만료 세션 케이스를 별도 확인한다.
- `REGRESSION_NOTES.md`의 인증 항목에 이번 체크를 링크한다.
```

이 저장 형식은 평가를 일회성 감상에서 팀의 품질 데이터로 바꾼다.
