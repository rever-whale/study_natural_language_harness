# 6. 작업 헌장: 목표, 경계, 완료 조건

작업 헌장은 자연어 하네스의 시작점이다. 이 문서는 AI에게 “무엇을 해줘”라고 말하기 전에, 사람과 팀이 이번 작업을 어떻게 정의하는지 고정한다.

작업 헌장이 없으면 모델은 요구사항을 확장하거나 축소한다. 때로는 좋은 의도로 불필요한 리팩터링을 하고, 때로는 중요한 예외 케이스를 누락한다.

## 작업 헌장의 구성

작업 헌장은 길 필요가 없다. 다음 항목이면 충분하다.

```text
작업 이름:

배경:

목표:

비목표:

대상 범위:

건드리면 안 되는 것:

완료 조건:

검증 방법:

보고 형식:
```

`비목표`는 특히 중요하다. 일반 개발 업무에서 AI가 일을 키우는 가장 흔한 이유는 하지 않아도 되는 일을 “개선”이라고 판단하기 때문이다.

## 완료 조건을 테스트처럼 쓰기

코드를 쓰지 않더라도 완료 조건은 테스트처럼 써야 한다.

```text
완료 조건:
- 신규 사용자는 이메일 인증 없이 대시보드에 진입할 수 없다.
- 기존 로그인 사용자는 세션이 유지되는 동안 추가 인증을 요구받지 않는다.
- 인증 실패 메시지는 기존 문구를 유지한다.
- 변경 사항은 회원가입 플로우 밖으로 확산되지 않는다.
```

이렇게 쓰면 AI도, 리뷰어도 같은 기준으로 결과를 본다.

## 작업 헌장을 AI에게 검토시키기

작업 헌장은 사람이 초안을 쓰고 AI에게 바로 수행을 맡기기 전에 검토시킨다.

```text
다음 작업 헌장을 읽고 바로 작업하지 말아줘.
먼저 모호한 용어, 빠진 완료 조건, 위험한 범위 확장을 찾아줘.
확인 질문은 최대 7개로 제한하고, 질문마다 왜 중요한지 한 줄로 설명해줘.
```

이 단계는 Ouroboros식 명세 우선 접근을 팀 문서로 옮긴 것이다. 코드를 쓰기 전에 무엇을 만들지 더 분명하게 만드는 데 시간을 쓴다.

## 작업 헌장 저장 위치

프로젝트 지식베이스에서는 작업 헌장을 다음처럼 보관한다.

```text
ai-harness/
  tasks/
    2026-06-auth-email-verification.md
    2026-06-dashboard-empty-state.md
```

완료된 작업도 삭제하지 않는다. 나중에 비슷한 작업을 할 때 좋은 예시와 회귀 체크가 된다.
