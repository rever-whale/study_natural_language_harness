# 1. 왜 이번 책은 코드를 쓰지 않는가

하네스라는 말을 들으면 보통 런타임, 도구 어댑터, 평가 파이프라인, 트레이스 저장소 같은 구현물을 떠올린다. 그런 구현은 중요하지만, 일반 애플리케이션 개발팀이 AI를 실무에 들여놓을 때 처음 막히는 지점은 대개 코드가 아니다. 문제는 팀이 AI에게 일을 맡기는 방식이 문서화되어 있지 않다는 데 있다.

AI 작업은 흔히 다음 흐름으로 무너진다.

1. 누군가 모델에게 막연한 요청을 던진다.
2. 모델이 그럴듯한 결과를 만든다.
3. 사람이 맥락 누락, 잘못된 가정, 검증 누락을 뒤늦게 발견한다.
4. 같은 설명과 수정 요청이 다음 세션에서 반복된다.
5. 팀은 “AI는 빠른데 믿기는 어렵다”는 결론에 가까워진다.

이 책의 접근은 반대다. 모델을 더 정교하게 제어하는 코드를 먼저 만들지 않는다. 대신 팀이 이미 쓰는 자연어 문서, 이슈, PR 설명, ADR, 회고, 체크리스트를 하네스의 재료로 삼는다.

## 자연어 하네스의 정의

자연어 하네스는 AI 에이전트가 특정 프로젝트에서 일관되게 행동하도록 돕는 문서 기반 실행 환경이다. 여기서 자연어는 그냥 긴 설명이 아니다. 팀이 합의한 구조, 이름, 순서, 판정 기준을 가진 운영 인터페이스다.

자연어 하네스는 다음 질문에 답한다.

- 이 프로젝트에서 AI에게 맡겨도 되는 일과 맡기면 안 되는 일은 무엇인가?
- 작업을 시작하기 전에 어떤 질문을 반드시 해야 하는가?
- 에이전트가 읽어야 할 최소 프로젝트 맥락은 무엇인가?
- 결과물을 어떤 순서로 검증해야 하는가?
- 실패와 교훈은 어디에 기록하고 다음 세션에 어떻게 반영하는가?

이 질문들에 대한 답이 없으면, 어떤 모델을 써도 팀의 AI 활용은 개인기 위에 머문다.

## 코드 없는 접근이 유리한 순간

자연어 하네스는 특히 다음 상황에서 유리하다.

- 팀이 아직 에이전트 런타임을 직접 만들 계획이 없다.
- Claude, Codex, Copilot Chat처럼 이미 완성된 도구를 로컬에서 사용한다.
- 프로젝트마다 업무 방식과 제약이 달라 하드코딩된 도구보다 문서 규칙이 더 빠르다.
- 보안상 외부 SaaS 평가 파이프라인을 쓰기 어렵다.
- 조직 안에서 먼저 합의해야 할 것은 기술보다 작업 방식이다.

코드가 필요 없다는 뜻은 아니다. 나중에 충분히 반복되는 문서 절차는 자동화할 수 있다. 다만 처음부터 자동화를 목표로 하면, 아직 검증되지 않은 운영 방식을 코드로 굳혀버릴 위험이 있다.

## 이 책의 산출물

이 책을 따라가며 팀은 다음 산출물을 만든다.

- `PROJECT_CONTEXT.md`: 프로젝트의 목적, 구조, 실행 제약, 용어를 요약한다.
- `AI_WORKFLOW.md`: AI에게 일을 맡기는 표준 절차를 정의한다.
- `PROMPT_LIBRARY.md`: 검증된 프롬프트 패턴을 저장한다.
- `EVAL_RUBRIC.md`: 결과물을 평가하는 기준을 정한다.
- `SESSION_LOG.md`: AI 세션의 목표, 결정, 검증, 후속 작업을 기록한다.
- `HARNESS_CHANGELOG.md`: 하네스 자체의 변경 이력을 남긴다.

파일 이름은 예시다. 중요한 것은 특정 형식이 아니라 팀이 같은 구조를 반복 사용하는 것이다.
