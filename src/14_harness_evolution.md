# 14. 하네스를 디벨롭하는 방법

자연어 하네스는 한 번 만들고 끝나는 문서가 아니다. 프로젝트가 변하고 팀의 AI 사용 방식이 성숙하면 하네스도 변해야 한다. 중요한 것은 변경을 우발적으로 하지 않는 것이다.

## 하네스 변경의 입력

하네스는 다음 신호를 바탕으로 바꾼다.

- 같은 질문을 여러 세션에서 반복한다.
- 같은 실수가 두 번 이상 재발한다.
- 팀원이 프롬프트 라이브러리를 쓰지 않는다.
- 평가 루브릭이 너무 추상적이라 판정이 흔들린다.
- 프로젝트 구조가 바뀌었는데 컨텍스트 문서가 따라오지 못한다.
- AI가 자꾸 범위를 넓히거나 엉뚱한 문서를 참조한다.

이런 신호는 하네스의 버그 리포트다.

## 하네스 버그 분류

하네스 문제를 그냥 “문서가 부족하다”로 보면 해결하기 어렵다. 다음처럼 분류한다.

| 유형 | 증상 | 대응 |
| --- | --- | --- |
| Context Gap | AI가 같은 프로젝트 맥락을 반복 질문 | `PROJECT_CONTEXT.md` 또는 컨텍스트 패킷 갱신 |
| Ambiguity Gap | 핵심 용어를 다르게 해석 | `GLOSSARY.md`와 작업 헌장 템플릿 보강 |
| Boundary Gap | AI가 범위를 넓힘 | 비목표, 금지 범위, 권한 규칙 강화 |
| Evaluation Gap | 결과가 좋아 보이지만 판정 어려움 | `EVAL_RUBRIC.md` 상세화 |
| Regression Gap | 같은 실패가 반복 | `REGRESSION_NOTES.md` 승격 |
| Adoption Gap | 팀원이 하네스를 쓰지 않음 | 템플릿 축소, 시작 절차 단순화 |

분류가 있으면 하네스 개선이 감정적 회고에서 운영 작업으로 바뀐다.

## 하네스 변경 절차

```text
HARNESS_CHANGELOG 항목:

날짜:
변경자:
변경 대상:
문제:
변경 내용:
기대 효과:
검증 방법:
되돌릴 조건:
```

하네스 문서도 운영 시스템이다. 변경 이유가 없으면 나중에 왜 그런 규칙이 생겼는지 알 수 없다.

## 개선 실험 설계

하네스 변경은 작은 실험으로 다룬다.

```text
실험 이름:
반복되는 문제:
변경할 하네스 요소:
기대하는 AI 행동 변화:
관찰 기간:
성공 기준:
실패 시 되돌릴 방법:
```

예를 들어 “AI가 인증 작업에서 만료 세션과 권한 없음 상태를 혼동한다”는 문제가 있다면, `REGRESSION_NOTES.md`에 체크를 추가하고 2주 동안 인증 관련 세션에서 그 체크가 실제로 사용되는지 본다.

하네스 개선은 문서 작성량으로 평가하지 않는다. 다음 세션에서 실제 행동이 좋아졌는지로 평가한다.

## 좋은 하네스 개선의 예

나쁜 개선은 문서를 길게 만드는 것이다. 좋은 개선은 다음 중 하나를 달성한다.

- 반복 설명을 줄인다.
- 질문의 질을 높인다.
- 위험한 작업을 더 빨리 드러낸다.
- 검증을 더 구체적으로 만든다.
- 오래된 정보를 더 쉽게 제거한다.
- 새 팀원이 더 빨리 같은 품질의 세션을 시작하게 한다.

문서가 길어졌는데 이 중 아무것도 좋아지지 않았다면 개선이 아닐 수 있다.

## 자동화로 넘어가는 기준

자연어 하네스의 일부는 나중에 자동화할 수 있다. 다만 다음 조건이 충족될 때가 좋다.

- 같은 절차가 최소 5회 이상 반복되었다.
- 사람이 수동으로 해도 가치가 있다는 것이 확인되었다.
- 입력과 출력 형식이 안정되었다.
- 실패했을 때 사람이 쉽게 개입할 수 있다.
- 자동화가 팀의 이해를 가리지 않는다.

먼저 문서로 운영하고, 그다음 반복되는 부분을 도구화한다. 이 순서가 팀에 더 안전하다.

## 자동화 후보 예시

자연어 하네스가 안정되면 다음을 자동화할 수 있다.

- 새 작업 생성 시 작업 헌장 템플릿 자동 생성
- 세션 로그에서 `하네스 문서에 반영할 개선점`만 모아 주간 리뷰 안건 생성
- `deprecated` 문서가 컨텍스트 패킷에 들어가면 경고
- 평가 결과에서 `Unverified` 항목이 있으면 PR 체크에 표시
- `REGRESSION_NOTES.md`의 관련 항목을 작업 유형별로 추천

하지만 자동화는 하네스를 대체하지 않는다. 자동화는 팀이 이미 이해하고 합의한 절차를 덜 귀찮게 만드는 단계다.
