# 학습 개요

이 책은 AI 에이전트나 프롬프트 하네스를 직접 구현하는 엔지니어를 위한 책이 아니다. 일반적인 애플리케이션 개발자가 Claude, Codex, Copilot 계열 로컬 세션을 더 안정적으로 쓰기 위해, 코드가 아니라 자연어 문서와 팀 운영 규칙으로 하네스를 구축하는 방법을 다룬다.

이전 책이 `Spec -> Execute -> Evaluate` 루프를 TypeScript 코드로 구현하는 방향이었다면, 이번 책은 같은 사상을 팀의 문서 체계로 옮긴다. 여기서 하네스는 실행 프로그램이 아니라 다음 문서 묶음이다.

- 작업 헌장: 무엇을 만들고, 무엇을 만들지 않을지 정한다.
- 인터뷰 노트: 숨은 가정과 애매한 단어를 드러낸다.
- 컨텍스트 패킷: 에이전트가 매번 읽어야 하는 프로젝트 지식을 압축한다.
- 실행 장부: 계획, 결정, 검증, 남은 위험을 기록한다.
- 평가 루브릭: 결과물이 충분히 좋은지 팀 기준으로 판정한다.
- 회귀 체크: 예전에 깨졌던 약속이 다시 깨지지 않게 한다.

핵심 전제는 단순하다. 모델에게 매번 긴 설명을 새로 하는 팀은 같은 실패를 반복한다. 반대로 팀이 프롬프트와 하네스를 지식베이스로 관리하면, 좋은 작업 방식이 개인의 감각이 아니라 프로젝트 자산이 된다.

## 학습 목표

이 책을 끝까지 읽으면 다음을 할 수 있다.

- 프롬프트를 일회성 문장이 아니라 재사용 가능한 작업 인터페이스로 설계한다.
- 코드 없이 자연어 문서만으로 에이전트 세션의 입력, 실행, 검증, 기록 흐름을 만든다.
- 프로젝트별로 `작업 헌장`, `컨텍스트 패킷`, `평가 루브릭`, `세션 장부`를 운영한다.
- 팀이 함께 쓰는 하네스 지식베이스를 만들고, 실제 프로젝트 피드백으로 발전시킨다.
- Claude나 Codex 같은 로컬 에이전트에게 같은 품질의 작업 환경을 반복 제공한다.

## 대상 독자

주 독자는 애플리케이션 개발자다. 백엔드, 프런트엔드, 모바일, 데이터, DevOps 어느 쪽이든 상관없다. 다만 다음 상황을 이미 경험한 사람에게 특히 유용하다.

- AI에게 일을 시키면 처음에는 빨라 보이지만, 뒤로 갈수록 검토 비용이 커진다.
- 팀원이 각자 다른 방식으로 프롬프트를 쓰기 때문에 결과 품질이 흔들린다.
- 프로젝트 맥락을 매번 설명하다가 세션이 길어지고, 중요한 제약은 자주 빠진다.
- AI가 만든 결과를 어떻게 검증하고 기록해야 할지 팀 기준이 없다.
- 한 프로젝트에서 잘 먹힌 프롬프트를 다른 프로젝트로 옮기기 어렵다.

## 책의 구조

Part 1은 프롬프팅을 다룬다. 다만 연구용 기법을 나열하지 않고, 애플리케이션 개발자가 실제 업무에서 쓸 수 있는 패턴으로 정리한다.

Part 2는 자연어 하네스의 부품을 만든다. Ouroboros의 명세 우선 사고를 코드 없는 팀 문서 체계로 바꾸어, 인터뷰, 명세, 실행, 평가, 기록의 순환 구조를 설계한다.

Part 3은 운영이다. 지식베이스를 프로젝트별로 어떻게 나누고, 팀 리뷰를 어떻게 돌리고, 시간이 지나며 하네스를 어떻게 개선할지 다룬다.

Part 4는 실전 적용 사례다. 첫 `ai-harness/`를 만드는 방법, 인증 버그 분석 세션, 신규 기능 명세 세션, 팀 거버넌스, 성숙도 모델을 통해 자연어 하네스가 실제 프로젝트에서 어떻게 자라고 관리되는지 보여준다.

Part 5는 책 자체를 하네스로 운영하는 방법이다. `_state` 상태 장부에 진행 상황, 자체 리뷰, 품질 게이트, 보강 반복 로그를 남기고, 충분한 수준에 도달할 때까지 문서를 다시 검토하고 확장하는 과정을 다룬다.

부록에는 Claude나 Codex에게 바로 붙여 넣어 쓸 수 있는 자연어 템플릿, 소스 맵, 용어집을 둔다.
