17. 사례 연구: 인증 버그 수정 세션

이 장은 자연어 하네스를 실제 작업에 적용하는 완성 사례다. 사례는 가상의 관리자 콘솔 프로젝트를 기준으로 하지만, 흐름은 대부분의 애플리케이션 개발팀에 그대로 적용할 수 있다.

시작 요청

사용자의 첫 요청은 보통 이렇게 들어온다.

로그인 풀리는 버그 좀 봐줘. 가끔 관리자 페이지에서 빈 화면이 떠.

이 요청은 바로 실행하기에 위험하다. “로그인 풀림”, “가끔”, “관리자 페이지”, “빈 화면”이 모두 모호하다. 자연어 하네스는 이 요청을 작업 헌장으로 바꾸는 것부터 시작한다.

인터뷰

AI에게 먼저 인터뷰를 시킨다.

너는 이 프로젝트의 명세 인터뷰어다.
아직 해결책을 제안하지 말고, 아래 요청의 숨은 가정과 모호한 용어를 드러내줘.

요청:
로그인 풀리는 버그 좀 봐줘. 가끔 관리자 페이지에서 빈 화면이 떠.

좋은 응답은 다음 질문을 포함한다.

  • “로그인 풀림”은 세션 만료, 토큰 갱신 실패, 권한 없음 중 무엇인가?
  • 빈 화면은 특정 URL에서만 발생하는가?
  • 사용자는 workspace admin인가, 일반 member인가?
  • 재현 가능한 브라우저나 시간 조건이 있는가?
  • 최근 인증, 라우팅, 권한 관련 변경이 있었는가?
  • 기대 동작은 로그인 페이지 이동인가, 에러 화면인가, 세션 갱신인가?

이 질문은 느려 보이지만, 잘못된 수정을 막는다.

작업 헌장

인터뷰 답변을 바탕으로 작업 헌장을 만든다.

# 작업 헌장: 관리자 콘솔 빈 화면 인증 버그 분석

배경:
일부 workspace admin이 관리자 페이지 진입 시 빈 화면을 본다는 제보가 있다.
발생 시점은 세션 만료 이후 재접속으로 추정된다.

목표:
빈 화면이 세션 만료, 권한 없음, 라우팅 오류 중 무엇에서 발생하는지 구분하고,
재현 조건과 수정 방향을 제시한다.

비목표:
- 인증 정책을 변경하지 않는다.
- 권한 모델을 재설계하지 않는다.
- 사용자 노출 문구를 확정하지 않는다.
- 바로 배포 가능한 코드를 작성하지 않는다.

대상 범위:
- 관리자 콘솔 진입 라우팅
- 세션 상태 확인 흐름
- 권한 가드
- 에러/리다이렉트 처리

완료 조건:
- 세션 만료, 권한 없음, 게스트 접근의 현재 동작이 분리되어 설명된다.
- 빈 화면을 만드는 가장 가능성 높은 원인 후보가 정리된다.
- 각 후보의 확인 방법이 제시된다.
- 수정 작업이 필요하면 후속 작업 헌장 초안이 작성된다.

검증 방법:
- PROJECT_CONTEXT.md의 인증/권한 규칙 확인
- REGRESSION_NOTES.md의 인증 항목 확인
- 관련 세션 로그나 과거 이슈 확인

컨텍스트 패킷

이 작업의 컨텍스트 패킷은 다음처럼 구성한다.

반드시 읽기:
- ai-harness/PROJECT_CONTEXT.md
- ai-harness/GLOSSARY.md
- ai-harness/REGRESSION_NOTES.md
- ai-harness/tasks/2026-06-auth-empty-screen.md

필요하면 읽기:
- ai-harness/decisions/2026-03-auth-routing.md
- ai-harness/sessions/2026-05-28-role-guard.md

읽지 말기:
- docs/old-auth-flow.md: 2025년 인증 정책 문서로 현재 IdP 연동 전 내용

주의:
- 인증 실패 문구는 고객지원 매크로와 연결되어 있어 임의 변경 금지
- 권한 없음과 세션 만료를 같은 상태로 처리하지 말 것

컨텍스트 패킷은 에이전트가 탐색할 방향을 좁힌다. 특히 “읽지 말기”가 중요하다. 낡은 문서는 없는 것보다 위험할 수 있다.

로컬 세션 프롬프트

Codex나 Claude 세션에서는 다음처럼 진행한다.

이번 세션 모드: Read-only Diagnosis

규칙:
- 아직 파일을 수정하지 않는다.
- 작업 헌장과 컨텍스트 패킷을 먼저 읽는다.
- 세션 만료, 권한 없음, 게스트 접근을 구분해서 현재 동작을 설명한다.
- 원인 후보마다 확인 방법과 위험도를 제시한다.
- 마지막에 평가 루브릭 기준으로 미확인 항목을 표시한다.

읽기 전용 진단 모드로 시작하면 AI가 성급하게 코드를 바꾸는 것을 막을 수 있다.

기대되는 결과

좋은 AI 결과는 다음 구조를 가진다.

## 관찰 요약

- 게스트 접근은 로그인 페이지로 이동한다.
- 권한 없음은 403 상태를 받아야 하지만 현재 일부 경로에서 빈 화면으로 남을 수 있다.
- 만료 세션은 토큰 갱신 실패 후 리다이렉트 처리가 누락될 가능성이 있다.

## 원인 후보

1. 세션 갱신 실패 후 라우터 fallback 미처리
   - 가능성: 높음
   - 확인: 만료 세션 상태에서 admin route 진입 흐름 확인

2. 권한 없음 응답을 빈 상태와 같은 분기로 처리
   - 가능성: 중간
   - 확인: role guard의 403 처리 확인

3. 오래된 인증 문서 기반의 잘못된 가정
   - 가능성: 낮음
   - 확인: 현재 PROJECT_CONTEXT와 old-auth-flow 차이 확인

## 후속 작업 제안

- 만료 세션 처리 수정 작업 헌장 작성
- 권한 없음과 세션 만료 용어를 GLOSSARY.md에 분리 정의
- REGRESSION_NOTES.md에 인증 상태 혼동 체크 추가

이 결과가 좋은 이유는 바로 수정을 하지 않았기 때문이 아니다. 문제 공간을 분리하고, 후속 수정이 검증 가능하도록 만들었기 때문이다.

평가

평가 루브릭으로 결과를 판정한다.

Overall: Needs Review

완료 조건별 판정:
- 세션 만료, 권한 없음, 게스트 접근 분리 설명: Pass
- 원인 후보 정리: Pass
- 후보별 확인 방법 제시: Pass
- 후속 작업 헌장 초안 작성: Unverified

남은 위험:
- 실제 코드/로그 확인 전이므로 원인 확정은 아님.
- 고객지원 문구와 연결된 에러 메시지 정책은 추가 확인 필요.

여기서 Needs Review는 실패가 아니다. 자연어 하네스에서는 “어디까지 확인했고 어디부터 사람이 봐야 하는지”가 드러나면 좋은 결과다.

하네스 업데이트

세션이 끝나면 지식베이스를 업데이트한다.

GLOSSARY.md 추가:

- expired session: 사용자의 인증 세션이 만료되어 재인증이 필요한 상태.
- unauthorized: 인증되지 않은 사용자가 보호된 경로에 접근한 상태.
- forbidden: 인증은 되었지만 해당 리소스 권한이 없는 상태.

REGRESSION_NOTES.md 추가:

## 인증 상태 혼동

- 문제: 만료 세션, 인증 없음, 권한 없음이 같은 UI 실패처럼 보이는 세션이 반복됨.
- 체크: 인증 관련 작업은 expired session, unauthorized, forbidden을 분리해 검증한다.
- 근거: sessions/2026-06-06-auth-empty-screen.md

이 업데이트까지 포함되어야 한 사이클이 끝난다. AI가 준 답변만 받고 문서를 바꾸지 않으면, 다음 세션은 같은 질문을 다시 하게 된다.
