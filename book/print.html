자연어 하네스와 팀 프롬프팅

학습 개요

이 책은 AI 에이전트나 프롬프트 하네스를 직접 구현하는 엔지니어를 위한 책이 아니다. 일반적인 애플리케이션 개발자가 Claude, Codex, Copilot 계열 로컬 세션을 더 안정적으로 쓰기 위해, 코드가 아니라 자연어 문서와 팀 운영 규칙으로 하네스를 구축하는 방법을 다룬다.

이전 책이 Spec -> Execute -> Evaluate 루프를 TypeScript 코드로 구현하는 방향이었다면, 이번 책은 같은 사상을 팀의 문서 체계로 옮긴다. 여기서 하네스는 실행 프로그램이 아니라 다음 문서 묶음이다.

  • 작업 헌장: 무엇을 만들고, 무엇을 만들지 않을지 정한다.
  • 인터뷰 노트: 숨은 가정과 애매한 단어를 드러낸다.
  • 컨텍스트 패킷: 에이전트가 매번 읽어야 하는 프로젝트 지식을 압축한다.
  • 실행 장부: 계획, 결정, 검증, 남은 위험을 기록한다.
  • 평가 루브릭: 결과물이 충분히 좋은지 팀 기준으로 판정한다.
  • 회귀 체크: 예전에 깨졌던 약속이 다시 깨지지 않게 한다.

핵심 전제는 단순하다. 모델에게 매번 긴 설명을 새로 하는 팀은 같은 실패를 반복한다. 반대로 팀이 프롬프트와 하네스를 지식베이스로 관리하면, 좋은 작업 방식이 개인의 감각이 아니라 프로젝트 자산이 된다.

학습 목표

이 책을 끝까지 읽으면 다음을 할 수 있다.

  • 프롬프트를 일회성 문장이 아니라 재사용 가능한 작업 인터페이스로 설계한다.
  • 코드 없이 자연어 문서만으로 에이전트 세션의 입력, 실행, 검증, 기록 흐름을 만든다.
  • 프로젝트별로 작업 헌장, 컨텍스트 패킷, 평가 루브릭, 세션 장부를 운영한다.
  • 팀이 함께 쓰는 하네스 지식베이스를 만들고, 실제 프로젝트 피드백으로 발전시킨다.
  • Claude나 Codex 같은 로컬 에이전트에게 같은 품질의 작업 환경을 반복 제공한다.

대상 독자

주 독자는 애플리케이션 개발자다. 백엔드, 프런트엔드, 모바일, 데이터, DevOps 어느 쪽이든 상관없다. 다만 다음 상황을 이미 경험한 사람에게 특히 유용하다.

  • AI에게 일을 시키면 처음에는 빨라 보이지만, 뒤로 갈수록 검토 비용이 커진다.
  • 팀원이 각자 다른 방식으로 프롬프트를 쓰기 때문에 결과 품질이 흔들린다.
  • 프로젝트 맥락을 매번 설명하다가 세션이 길어지고, 중요한 제약은 자주 빠진다.
  • AI가 만든 결과를 어떻게 검증하고 기록해야 할지 팀 기준이 없다.
  • 한 프로젝트에서 잘 먹힌 프롬프트를 다른 프로젝트로 옮기기 어렵다.

책의 구조

Part 1은 프롬프팅을 다룬다. 다만 연구용 기법을 나열하지 않고, 애플리케이션 개발자가 실제 업무에서 쓸 수 있는 패턴으로 정리한다.

Part 2는 자연어 하네스의 부품을 만든다. Ouroboros의 명세 우선 사고를 코드 없는 팀 문서 체계로 바꾸어, 인터뷰, 명세, 실행, 평가, 기록의 순환 구조를 설계한다.

Part 3은 운영이다. 지식베이스를 프로젝트별로 어떻게 나누고, 팀 리뷰를 어떻게 돌리고, 시간이 지나며 하네스를 어떻게 개선할지 다룬다.

Part 4는 실전 적용 사례다. 첫 ai-harness/를 만드는 방법, 인증 버그 분석 세션, 신규 기능 명세 세션, 팀 거버넌스, 성숙도 모델을 통해 자연어 하네스가 실제 프로젝트에서 어떻게 자라고 관리되는지 보여준다.

Part 5는 책 자체를 하네스로 운영하는 방법이다. _state 상태 장부에 진행 상황, 자체 리뷰, 품질 게이트, 보강 반복 로그를 남기고, 충분한 수준에 도달할 때까지 문서를 다시 검토하고 확장하는 과정을 다룬다.

부록에는 Claude나 Codex에게 바로 붙여 넣어 쓸 수 있는 자연어 템플릿, 소스 맵, 용어집을 둔다.

1. 왜 이번 책은 코드를 쓰지 않는가

하네스라는 말을 들으면 보통 런타임, 도구 어댑터, 평가 파이프라인, 트레이스 저장소 같은 구현물을 떠올린다. 그런 구현은 중요하지만, 일반 애플리케이션 개발팀이 AI를 실무에 들여놓을 때 처음 막히는 지점은 대개 코드가 아니다. 문제는 팀이 AI에게 일을 맡기는 방식이 문서화되어 있지 않다는 데 있다.

AI 작업은 흔히 다음 흐름으로 무너진다.

  1. 누군가 모델에게 막연한 요청을 던진다.
  2. 모델이 그럴듯한 결과를 만든다.
  3. 사람이 맥락 누락, 잘못된 가정, 검증 누락을 뒤늦게 발견한다.
  4. 같은 설명과 수정 요청이 다음 세션에서 반복된다.
  5. 팀은 “AI는 빠른데 믿기는 어렵다”는 결론에 가까워진다.

이 책의 접근은 반대다. 모델을 더 정교하게 제어하는 코드를 먼저 만들지 않는다. 대신 팀이 이미 쓰는 자연어 문서, 이슈, PR 설명, ADR, 회고, 체크리스트를 하네스의 재료로 삼는다.

자연어 하네스의 정의

자연어 하네스는 AI 에이전트가 특정 프로젝트에서 일관되게 행동하도록 돕는 문서 기반 실행 환경이다. 여기서 자연어는 그냥 긴 설명이 아니다. 팀이 합의한 구조, 이름, 순서, 판정 기준을 가진 운영 인터페이스다.

자연어 하네스는 다음 질문에 답한다.

  • 이 프로젝트에서 AI에게 맡겨도 되는 일과 맡기면 안 되는 일은 무엇인가?
  • 작업을 시작하기 전에 어떤 질문을 반드시 해야 하는가?
  • 에이전트가 읽어야 할 최소 프로젝트 맥락은 무엇인가?
  • 결과물을 어떤 순서로 검증해야 하는가?
  • 실패와 교훈은 어디에 기록하고 다음 세션에 어떻게 반영하는가?

이 질문들에 대한 답이 없으면, 어떤 모델을 써도 팀의 AI 활용은 개인기 위에 머문다.

코드 없는 접근이 유리한 순간

자연어 하네스는 특히 다음 상황에서 유리하다.

  • 팀이 아직 에이전트 런타임을 직접 만들 계획이 없다.
  • Claude, Codex, Copilot Chat처럼 이미 완성된 도구를 로컬에서 사용한다.
  • 프로젝트마다 업무 방식과 제약이 달라 하드코딩된 도구보다 문서 규칙이 더 빠르다.
  • 보안상 외부 SaaS 평가 파이프라인을 쓰기 어렵다.
  • 조직 안에서 먼저 합의해야 할 것은 기술보다 작업 방식이다.

코드가 필요 없다는 뜻은 아니다. 나중에 충분히 반복되는 문서 절차는 자동화할 수 있다. 다만 처음부터 자동화를 목표로 하면, 아직 검증되지 않은 운영 방식을 코드로 굳혀버릴 위험이 있다.

이 책의 산출물

이 책을 따라가며 팀은 다음 산출물을 만든다.

  • PROJECT_CONTEXT.md: 프로젝트의 목적, 구조, 실행 제약, 용어를 요약한다.
  • AI_WORKFLOW.md: AI에게 일을 맡기는 표준 절차를 정의한다.
  • PROMPT_LIBRARY.md: 검증된 프롬프트 패턴을 저장한다.
  • EVAL_RUBRIC.md: 결과물을 평가하는 기준을 정한다.
  • SESSION_LOG.md: AI 세션의 목표, 결정, 검증, 후속 작업을 기록한다.
  • HARNESS_CHANGELOG.md: 하네스 자체의 변경 이력을 남긴다.

파일 이름은 예시다. 중요한 것은 특정 형식이 아니라 팀이 같은 구조를 반복 사용하는 것이다.

2. 프롬프트는 요청이 아니라 작업 인터페이스다

프롬프트를 “질문을 잘 쓰는 법”으로만 보면 실무에서는 금방 한계가 온다. 애플리케이션 개발에서 프롬프트는 작업 인터페이스다. 사람과 AI가 같은 작업을 바라보는 입구이며, 입력 계약과 출력 계약을 함께 담아야 한다.

좋은 작업 인터페이스는 친절한 문장이 아니라 반복 가능한 구조다.

  • 역할: 지금 AI는 설계자, 리뷰어, 구현 보조자, 문서 편집자 중 무엇인가?
  • 목표: 이번 세션의 성공 상태는 무엇인가?
  • 입력: AI가 반드시 고려해야 할 자료는 무엇인가?
  • 제약: 건드리면 안 되는 파일, 지켜야 할 API, 금지된 접근은 무엇인가?
  • 절차: 먼저 읽고, 질문하고, 계획하고, 수행하고, 검증하는 순서는 어떻게 되는가?
  • 출력: 최종 응답은 어떤 형태여야 하며, 어떤 증거를 포함해야 하는가?

이 구조를 갖춘 프롬프트는 한 번의 좋은 답변을 얻기 위한 문장이 아니라, 팀원이 반복해서 사용할 수 있는 작업 도구가 된다.

애플리케이션 개발자의 기본 프롬프트 구조

실무 프롬프트는 보통 다음 네 단락이면 충분하다.

상황:
이 프로젝트의 현재 맥락과 작업 배경을 설명한다.

목표:
이번 세션이 끝났을 때 달성되어야 할 상태를 쓴다.

제약:
건드리지 말아야 할 것, 지켜야 할 스타일, 확인해야 할 위험을 쓴다.

진행 방식:
먼저 확인할 자료, 질문해야 할 조건, 수행 순서, 검증 방식을 쓴다.

이 형식은 단순하지만 강력하다. 특히 제약과 진행 방식이 빠지면 모델은 보통 가장 그럴듯한 길을 스스로 고른다. 그것이 팀의 실제 작업 방식과 다르면 결과물은 빠르게 빗나간다.

프롬프트 라이브러리의 단위

팀 지식베이스에 프롬프트를 저장할 때는 “좋았던 문장”을 그대로 모으지 않는다. 다음 단위로 분리한다.

  • 목적: 이 프롬프트가 해결하는 작업 유형
  • 사용 조건: 언제 쓰면 좋고, 언제 쓰면 안 되는지
  • 입력 슬롯: 사용자가 채워야 하는 변수
  • 기대 출력: 결과 형식과 품질 기준
  • 실패 징후: 모델이 흔히 잘못 가는 방향
  • 개선 이력: 왜 문구가 바뀌었는지

프롬프트는 작은 API처럼 다루는 편이 좋다. 이름, 입력, 출력, 실패 모드가 있어야 팀원이 재사용할 수 있다.

자연어 계약으로 쓰기

프롬프트의 마지막에는 짧은 계약 문장을 둔다.

위 조건이 모호하면 바로 작업하지 말고, 먼저 최대 5개의 확인 질문을 해줘.
확인 질문 없이 진행할 수 있다면, 계획을 5줄 이내로 제시한 뒤 작업을 시작해줘.
최종 응답에는 변경 사항, 검증 결과, 남은 위험을 분리해서 알려줘.

이 문장은 모델을 완벽하게 통제하지는 못한다. 하지만 팀이 원하는 상호작용 방식을 반복 노출한다. 반복 노출은 자연어 하네스에서 중요한 힘이다.

3. 일반 개발자를 위한 프롬프팅 패턴

프롬프팅 기법은 많지만, 애플리케이션 개발자가 매일 쓰기 좋은 패턴은 비교적 정해져 있다. 이 장은 연구 기법 전체를 다루지 않고, 로컬 Claude나 Codex 세션에서 자주 쓰는 업무 패턴으로 다시 묶는다.

Zero-shot: 단순 업무의 기본값

Zero-shot은 예시 없이 바로 지시하는 방식이다. 간단한 요약, 오류 메시지 해석, 문서 초안, 작은 리팩터링 검토에 적합하다.

다음 에러 로그를 읽고, 원인 후보를 가능성 높은 순서로 정리해줘.
각 후보마다 확인 방법과 위험도를 함께 적어줘.

Zero-shot은 빠르지만, 팀 고유 기준을 반영하기 어렵다. 따라서 결과가 팀 스타일에 의존하는 작업에는 단독으로 쓰지 않는 편이 좋다.

Few-shot: 팀 기준을 예시로 주입하기

Few-shot은 좋은 입력과 출력 예시를 함께 제공하는 방식이다. 코드 리뷰 코멘트 스타일, 릴리즈 노트 문체, 장애 보고서 형식처럼 팀의 취향이 중요한 작업에 유용하다.

아래 예시의 밀도와 말투를 따라 PR 설명을 다듬어줘.

좋은 예시:
- 변경 이유:
- 사용자 영향:
- 검증:
- 배포 위험:

이번 PR 초안:
...

예시는 길 필요가 없다. 중요한 것은 모델이 흉내 낼 수 있는 판단 기준을 보여주는 것이다.

Chain-of-Thought 대신 공개 가능한 추론 요약

모델에게 내부 사고를 길게 쓰게 하는 방식은 도구와 정책에 따라 적절하지 않을 수 있다. 실무에서는 “자세한 사고 과정”보다 “결정 근거 요약”을 요구하는 편이 안전하고 읽기 쉽다.

최종 결론만 내지 말고, 결론에 영향을 준 핵심 근거 3가지를 요약해줘.
불확실한 부분은 추측이라고 표시해줘.

이 방식은 검토자가 결과를 빠르게 신뢰하거나 반박할 수 있게 한다.

Prompt Chaining: 한 번에 끝내지 않기

복잡한 작업은 하나의 거대한 프롬프트로 끝내려 하지 않는다. 단계를 나눈다.

  1. 맥락 읽기
  2. 모호한 점 질문
  3. 작업 계획 작성
  4. 작은 단위 실행
  5. 검증
  6. 회고와 지식베이스 반영

자연어 하네스에서는 이 체인을 AI_WORKFLOW.md에 고정한다. 팀원은 매번 새로 순서를 발명하지 않고, 작업 유형에 맞는 체인을 선택한다.

ReAct의 실무 번역

ReAct는 추론과 행동을 번갈아 수행하는 에이전트 패턴으로 알려져 있다. 코드를 직접 만들지 않는 팀에게는 다음 운영 규칙으로 번역하면 충분하다.

  • 관찰: 먼저 파일, 이슈, 로그, 문서를 읽는다.
  • 판단: 읽은 내용으로 다음 행동을 정한다.
  • 행동: 수정, 작성, 비교, 질문 중 하나를 수행한다.
  • 검증: 행동 결과를 확인한다.
  • 기록: 왜 그렇게 했는지 세션 장부에 남긴다.

즉, ReAct를 “생각을 많이 쓰는 프롬프트”로 이해하지 말고, 관찰 가능한 작업 루프로 이해한다.

RAG의 자연어 버전

팀이 별도 검색 시스템을 구축하지 않아도, 지식베이스를 잘 나누면 가벼운 RAG처럼 쓸 수 있다.

  • 프로젝트 개요는 PROJECT_CONTEXT.md에 둔다.
  • 도메인 용어는 GLOSSARY.md에 둔다.
  • 자주 하는 작업은 WORKFLOWS/에 둔다.
  • 검증 기준은 EVAL_RUBRIC.md에 둔다.
  • 과거 실패는 REGRESSION_NOTES.md에 둔다.

AI에게는 “이 작업 전에 어떤 문서를 읽어야 하는지”를 명확히 알려준다. 검색 자동화보다 먼저 필요한 것은 읽을 만한 문서의 구조다.

4. 실패하는 프롬프트의 공통 구조

프롬프트 실패는 모델의 지능 부족만으로 설명되지 않는다. 실무에서 반복되는 실패는 대부분 입력 계약이 약할 때 생긴다. 이 장에서는 팀이 지식베이스에 기록해야 할 실패 모드를 정리한다.

목표가 결과물이 아니라 활동으로 쓰인 경우

나쁜 목표는 “리팩터링해줘”, “문서화해줘”, “개선해줘”처럼 활동만 말한다. 좋은 목표는 완료 상태를 말한다.

나쁜 예:
이 컴포넌트를 개선해줘.

좋은 예:
이 컴포넌트에서 상태 계산과 렌더링 책임을 분리하고,
기존 테스트가 통과하며,
사용자에게 보이는 UI 텍스트와 동작은 바뀌지 않게 해줘.

목표가 활동이면 모델은 “무언가 했다”로 끝낸다. 목표가 상태이면 검증할 수 있다.

제약이 암묵지로 남은 경우

사람은 팀의 암묵적 규칙을 안다고 착각한다. 모델은 모른다. 예를 들어 “우리 프로젝트에서는 모달 안에 또 카드 UI를 넣지 않는다”, “이 API는 하위 호환 때문에 필드명을 바꾸면 안 된다” 같은 규칙은 매번 주입되거나 프로젝트 컨텍스트에 있어야 한다.

자연어 하네스의 첫 번째 임무는 암묵지를 명시지로 바꾸는 것이다.

입력 자료가 너무 많거나 너무 적은 경우

너무 적으면 모델은 빈칸을 추측한다. 너무 많으면 중요한 제약이 묻힌다. 컨텍스트 패킷은 “가능한 모든 자료”가 아니라 “이 작업에 필요한 최소 충분 자료”여야 한다.

작업 시작 전에 다음을 분리한다.

  • 반드시 읽을 자료
  • 필요하면 읽을 자료
  • 읽지 말아야 할 오래된 자료
  • 현재 믿을 수 없는 자료

특히 오래된 설계 문서는 위험하다. 지식베이스에는 문서의 최신성도 함께 기록해야 한다.

검증을 모델의 자기평가에 맡긴 경우

“잘 됐는지 확인해줘”는 약한 검증이다. 모델은 자기가 만든 결과를 낙관적으로 평가할 수 있다. 검증은 구체적이어야 한다.

검증 기준:
- 기존 사용자 흐름이 깨지지 않았는가?
- API 응답 필드가 바뀌지 않았는가?
- 접근성 라벨이 유지되는가?
- 에러 케이스가 문서화되었는가?
- 변경하지 않기로 한 파일을 건드리지 않았는가?

자연어 하네스는 검증 질문을 미리 준비해 둔다. 평가 기준이 있어야 결과물이 팀 기준에 닿는다.

세션의 배움이 저장되지 않는 경우

AI 세션에서 가장 아까운 손실은 결과물이 아니라 배움의 손실이다. 왜 실패했는지, 어떤 제약을 추가해야 하는지, 어떤 프롬프트가 효과 있었는지가 저장되지 않으면 다음 사람도 같은 비용을 낸다.

따라서 모든 중요한 세션은 마지막에 다음 세 줄을 남긴다.

이번 세션에서 새로 알게 된 프로젝트 사실:
다음 세션부터 컨텍스트에 추가해야 할 규칙:
하네스 문서에서 수정할 항목:

이 세 줄이 자연어 하네스를 성장시킨다.

5. 하네스란 무엇인가

하네스는 AI가 실제 작업을 수행할 때 주변을 둘러싸는 실행 환경이다. 코드 기반 하네스는 도구 호출, 권한, 메모리, 평가, 샌드박스, 트레이스를 프로그램으로 관리한다. 자연어 하네스는 같은 문제를 문서와 팀 절차로 다룬다.

둘의 목적은 같다.

  • 필요한 맥락을 제공한다.
  • 사용할 수 있는 행동을 제한한다.
  • 진행 상태를 보존한다.
  • 검증 루프를 만든다.
  • 실패에서 배워 다음 실행을 개선한다.

차이는 구현 방식이다. 자연어 하네스는 모든 것을 코드로 강제하지 않는다. 대신 사람이 읽고 수정할 수 있는 운영 문서로 에이전트 환경을 구성한다.

자연어 하네스의 다섯 계층

첫 번째 계층은 정체성이다. 프로젝트가 무엇이고, 어떤 품질을 중요하게 여기며, 어떤 결정을 이미 했는지 정리한다.

두 번째 계층은 입력이다. AI가 어떤 문서를 읽고 어떤 질문을 해야 하는지 정의한다.

세 번째 계층은 실행이다. 작업을 어떻게 쪼개고, 언제 멈추고, 어떤 순서로 검증할지 정한다.

네 번째 계층은 평가다. 결과물이 완료 조건을 만족하는지 판정한다.

다섯 번째 계층은 진화다. 하네스 자체를 변경하고, 변경 이유를 기록하고, 팀 기준으로 승인한다.

하네스가 아닌 것

하네스는 만능 프롬프트가 아니다. 한 문장으로 모든 작업을 잘하게 만드는 주문도 아니다. 또한 AI를 무조건 자유롭게 풀어놓는 것도 아니다.

좋은 하네스는 모델의 능력을 믿되, 프로젝트의 중요한 기준을 운에 맡기지 않는다.

애플리케이션 개발팀의 핵심 전환

일반 개발팀에게 가장 중요한 전환은 “AI에게 일을 설명한다”에서 “AI가 일할 환경을 설계한다”로 이동하는 것이다.

예를 들어 버그 수정을 맡길 때 단순히 에러를 붙여 넣는 대신 다음 환경을 제공한다.

  • 버그가 발생한 사용자 흐름
  • 관련된 코드 영역
  • 바뀌면 안 되는 계약
  • 재현과 검증 절차
  • 과거 유사 장애 기록
  • 최종 보고 형식

이 정도만 해도 모델의 작업은 크게 안정된다. 에이전트가 똑똑해서가 아니라, 작업 환경이 덜 애매하기 때문이다.

6. 작업 헌장: 목표, 경계, 완료 조건

작업 헌장은 자연어 하네스의 시작점이다. 이 문서는 AI에게 “무엇을 해줘”라고 말하기 전에, 사람과 팀이 이번 작업을 어떻게 정의하는지 고정한다.

작업 헌장이 없으면 모델은 요구사항을 확장하거나 축소한다. 때로는 좋은 의도로 불필요한 리팩터링을 하고, 때로는 중요한 예외 케이스를 누락한다.

작업 헌장의 구성

작업 헌장은 길 필요가 없다. 다음 항목이면 충분하다.

작업 이름:

배경:

목표:

비목표:

대상 범위:

건드리면 안 되는 것:

완료 조건:

검증 방법:

보고 형식:

비목표는 특히 중요하다. 일반 개발 업무에서 AI가 일을 키우는 가장 흔한 이유는 하지 않아도 되는 일을 “개선”이라고 판단하기 때문이다.

완료 조건을 테스트처럼 쓰기

코드를 쓰지 않더라도 완료 조건은 테스트처럼 써야 한다.

완료 조건:
- 신규 사용자는 이메일 인증 없이 대시보드에 진입할 수 없다.
- 기존 로그인 사용자는 세션이 유지되는 동안 추가 인증을 요구받지 않는다.
- 인증 실패 메시지는 기존 문구를 유지한다.
- 변경 사항은 회원가입 플로우 밖으로 확산되지 않는다.

이렇게 쓰면 AI도, 리뷰어도 같은 기준으로 결과를 본다.

작업 헌장을 AI에게 검토시키기

작업 헌장은 사람이 초안을 쓰고 AI에게 바로 수행을 맡기기 전에 검토시킨다.

다음 작업 헌장을 읽고 바로 작업하지 말아줘.
먼저 모호한 용어, 빠진 완료 조건, 위험한 범위 확장을 찾아줘.
확인 질문은 최대 7개로 제한하고, 질문마다 왜 중요한지 한 줄로 설명해줘.

이 단계는 Ouroboros식 명세 우선 접근을 팀 문서로 옮긴 것이다. 코드를 쓰기 전에 무엇을 만들지 더 분명하게 만드는 데 시간을 쓴다.

작업 헌장 저장 위치

7. 인터뷰와 명세화: Wonder에서 Ontology로

Ouroboros의 가장 유용한 아이디어는 “프롬프트를 더 잘 쓰자”가 아니라 “명세를 먼저 만들자”에 가깝다. 자연어 하네스에서는 이를 인터뷰와 명세화 절차로 바꾼다.

일반적인 AI 작업은 사용자의 첫 문장을 곧바로 실행하려 한다. 하지만 첫 문장은 대개 요구사항이 아니라 걱정, 불편, 추측, 욕망이 섞인 신호다. 좋은 하네스는 이 신호를 바로 실행하지 않고 질문으로 정리한다.

Wonder 질문

Wonder 질문은 넓히는 질문이다. 아직 답을 좁히지 않고 문제 공간을 탐색한다.

  • 왜 지금 이 작업이 필요한가?
  • 누가 이 변화의 영향을 받는가?
  • 사용자가 실제로 겪는 문제는 무엇인가?
  • 지금 떠올린 해결책 말고 다른 해결책은 있는가?
  • 하지 않으면 어떤 일이 생기는가?

Wonder 질문은 “무엇을 해야 하지?”를 풍부하게 만든다.

Ontology 질문

Ontology 질문은 단어의 뜻을 좁히는 질문이다. 팀이 같은 말을 같은 의미로 쓰도록 만든다.

  • 여기서 “사용자”는 익명 방문자, 가입자, 유료 고객 중 누구인가?
  • “완료”는 배포 완료인가, QA 완료인가, 문서 반영까지 포함하는가?
  • “성능 개선”은 로딩 시간, 서버 비용, 렌더링 안정성 중 무엇인가?
  • “관리자”는 어떤 권한을 가진 역할인가?
  • “실패”는 에러 응답인가, 빈 결과인가, 느린 응답인가?

Ontology 질문은 재작업을 줄인다. 단어가 선명해지면 모델이 채워 넣을 추측 공간이 줄어든다.

인터뷰 프롬프트

너는 이 프로젝트의 명세 인터뷰어다.
아직 해결책을 제안하지 말고, 내가 요청한 작업의 숨은 가정과 모호한 용어를 드러내줘.

진행 방식:
1. 먼저 Wonder 질문으로 문제 공간을 넓혀줘.
2. 그다음 Ontology 질문으로 핵심 용어를 좁혀줘.
3. 마지막에 답변이 필요한 질문을 우선순위 순서로 7개 이하로 정리해줘.

8. 컨텍스트 패킷과 프로젝트 지식베이스

컨텍스트 패킷은 AI 세션을 시작할 때 제공하는 최소 프로젝트 맥락이다. 지식베이스 전체를 던지는 것이 아니라, 이번 작업에 필요한 문서 묶음을 고르는 행위다.

자연어 하네스에서 컨텍스트 관리는 가장 현실적인 성능 개선 수단이다. 모델을 바꾸지 않아도, 입력 맥락의 품질을 높이면 결과가 크게 안정된다.

지식베이스 기본 구조

프로젝트 루트나 문서 저장소에 다음 구조를 둔다.

ai-harness/
  README.md
  PROJECT_CONTEXT.md
  GLOSSARY.md
  AI_WORKFLOW.md
  EVAL_RUBRIC.md
  REGRESSION_NOTES.md
  PROMPT_LIBRARY.md
  decisions/
  tasks/
  sessions/
  workflows/

README.md는 하네스 지식베이스의 입구다. AI에게도 사람에게도 “어디서 시작할지”를 알려준다.

PROJECT_CONTEXT.md

프로젝트 컨텍스트는 다음 질문에 답한다.

  • 이 프로젝트는 누구를 위해 무엇을 해결하는가?
  • 주요 사용자 흐름은 무엇인가?
  • 시스템 경계는 어디인가?
  • 핵심 도메인 용어는 무엇인가?
  • 바뀌면 위험한 계약은 무엇인가?
  • 실행, 테스트, 배포에서 주의할 점은 무엇인가?
  • 최신 정보와 오래된 정보는 어떻게 구분하는가?

이 문서는 길수록 좋은 문서가 아니다. AI가 매 세션 읽을 수 있어야 하므로 압축되어야 한다.

PROJECT_CONTEXT.md 완성 예시

다음은 실제 팀에서 시작점으로 쓸 수 있는 짧은 예시다. 핵심은 모든 정보를 담는 것이 아니라, 에이전트가 잘못 추측하면 위험한 정보를 먼저 고정하는 것이다.

# PROJECT_CONTEXT

상태: active
최종 검토일: 2026-06-06
소유자: platform-team

## 제품 목적

이 프로젝트는 B2B SaaS 관리자가 조직 멤버, 결제 상태, 접근 권한을 관리하는 웹 애플리케이션이다.
주요 사용자는 고객사의 workspace admin이며, 일반 멤버는 일부 읽기 화면만 접근한다.

## 시스템 경계

- 프런트엔드: React 기반 관리자 콘솔
- 백엔드: REST API
- 인증: 외부 IdP와 사내 세션 API를 함께 사용
- 결제: 별도 billing-service가 소유하므로 이 프로젝트에서 결제 상태를 직접 수정하지 않는다.

## 반드시 지킬 계약

- `workspaceId`는 URL, API 요청, 감사 로그에서 같은 의미로 사용한다.
- 인증 실패 문구는 고객지원 매크로와 연결되어 있어 임의 변경하지 않는다.
- 관리자 권한 변경은 감사 로그에 남아야 한다.

## 자주 깨지는 지점

- 게스트 사용자의 리다이렉트 루프
- 만료 세션과 권한 없음 상태의 혼동
- workspace admin과 organization owner의 권한 혼용

## AI 세션 주의

- 인증, 결제, 권한 변경은 반드시 작업 헌장을 먼저 작성한다.
- `REGRESSION_NOTES.md`의 인증/권한 항목을 확인하지 않은 결과는 통과로 보지 않는다.

이 정도의 문서는 모델에게 “프로젝트의 헌법”처럼 작동한다. 모든 구현 세부사항을 담지는 않지만, 잘못 건드리면 비용이 큰 경계를 선명하게 만든다.

컨텍스트 패킷 만들기

작업 시작 전 다음 형식으로 패킷을 만든다.

이번 작업의 컨텍스트 패킷:

반드시 읽기:
- PROJECT_CONTEXT.md
- tasks/2026-06-auth-email-verification.md
- EVAL_RUBRIC.md

필요하면 읽기:
- decisions/2026-03-auth-policy.md
- REGRESSION_NOTES.md

읽지 말기:
- docs/old-auth-flow.md: 2025년 정책이라 현재와 다름

주의:
- 인증 에러 문구는 고객지원 매크로와 연결되어 있어 임의 변경 금지

이렇게 쓰면 에이전트는 맥락을 덜 추측한다.

컨텍스트 패킷 선택 기준

컨텍스트 패킷은 많이 넣는 게임이 아니다. 다음 기준으로 고른다.

분류포함 기준예시
반드시 읽기완료 조건을 판단하는 데 직접 필요작업 헌장, 평가 루브릭, 관련 ADR
필요하면 읽기구현 선택이나 배경 확인에 도움과거 세션 로그, 유사 PR 설명
읽지 말기오래되었거나 현재 정책과 충돌폐기된 설계 문서, 이전 릴리즈 노트
질문하기문서로 판정할 수 없는 업무 판단정책 변경 승인 여부, 우선순위

가장 흔한 실수는 “관련 있어 보이는 자료”를 전부 넣는 것이다. 관련성보다 판정 가능성이 중요하다. 에이전트가 이번 작업에서 결정을 내리거나 검증할 때 실제로 써야 하는 자료만 우선한다.

지식베이스 운영 규칙

프로젝트 지식베이스는 다음 네 가지 규칙을 가져야 한다.

  • 모든 핵심 문서는 상태, 최종 검토일, 소유자, 적용 범위를 가진다.
  • deprecated 문서는 삭제하지 않더라도, AI가 읽지 않도록 이유를 표시한다.
  • 세션에서 새로 발견한 프로젝트 사실은 SESSION_LOG에 먼저 남기고, 주간 리뷰에서 핵심 문서로 승격한다.
  • 한 문서가 너무 커지면 PROJECT_CONTEXT.md에 요약만 남기고 세부 문서는 decisions/나 workflows/로 분리한다.

이 규칙은 문서가 시간이 지나며 “AI를 돕는 맥락”에서 “AI를 속이는 낡은 맥락”으로 변하는 것을 막는다.

문서 최신성 표시

각 핵심 문서 상단에는 다음 메타 정보를 둔다.

상태: active
최종 검토일: 2026-06-06
소유자: platform-team
적용 범위: checkout, billing, account
주의: legacy checkout v1에는 적용하지 않음

자연어 하네스의 위험 중 하나는 오래된 문서가 강한 권위를 갖는 것이다. 최신성 표시는 작지만 효과적인 방어 장치다.

컨텍스트 부채 관리

컨텍스트 부채는 문서가 많아서 생기는 문제가 아니라, 어떤 문서를 믿어야 하는지 알 수 없어서 생기는 문제다. 팀은 매주 다음 질문을 점검한다.

  • AI가 이번 주에 오래된 문서를 참조했는가?
  • PROJECT_CONTEXT.md에 없는 프로젝트 규칙을 사람이 반복 설명했는가?
  • 세션 로그에 남은 새 사실이 핵심 문서로 승격되지 않았는가?
  • 문서 소유자가 떠났거나 바뀌었는데 소유권이 갱신되지 않았는가?
  • 컨텍스트 패킷에 항상 들어가지만 실제로 쓰이지 않는 문서가 있는가?

이 질문에 답하면 지식베이스는 점점 짧고 강해진다. 좋은 자연어 하네스는 문서를 많이 쌓는 시스템이 아니라, 에이전트가 읽어야 할 맥락을 계속 정제하는 시스템이다.

9. 실행 루프: 계획, 수행, 검증, 기록

자연어 하네스의 실행 루프는 단순하다.

  1. 읽기
  2. 질문하기
  3. 계획하기
  4. 수행하기
  5. 검증하기
  6. 기록하기

중요한 것은 순서를 지키는 것이다. AI 세션은 급하게 수행 단계로 넘어갈수록 나중에 검토 비용이 커진다.

읽기

에이전트에게 먼저 읽을 문서를 지정한다.

먼저 컨텍스트 패킷의 "반드시 읽기" 문서를 읽고,
이번 작업에 직접 영향을 주는 제약만 요약해줘.
아직 변경이나 작성은 하지 말아줘.

이 단계에서 사람이 요약을 보고 맥락 누락을 발견할 수 있다.

질문하기

모호한 점이 있으면 질문을 요구한다.

작업을 진행하기 전에 완료 조건을 흔들 수 있는 질문만 해줘.
취향이나 사소한 구현 선택은 질문하지 말고 합리적으로 가정해도 된다.

질문도 하네스의 일부다. 질문이 너무 많으면 진행이 느려지고, 너무 적으면 추측이 늘어난다.

계획하기

계획은 길지 않아야 한다. 로컬 에이전트 세션에서는 짧고 검토 가능한 계획이 좋다.

실행 계획을 5단계 이내로 제시해줘.
각 단계마다 검증 방법을 함께 적어줘.

계획 단계에서 범위가 커지면 즉시 줄인다.

수행하기

수행 단계에서는 한 번에 너무 많은 일을 맡기지 않는다. 작업을 작게 나눈다.

  • 문서 초안 작성
  • 코드 변경 방향 제안
  • 리뷰 체크리스트 생성
  • 테스트 케이스 후보 작성
  • 릴리즈 노트 정리

자연어만 쓰는 하네스라도, 실제 프로젝트에서 에이전트가 코드를 수정할 수는 있다. 다만 이 책의 하네스 자체는 코드가 아니라 작업 통제 문서라는 점이 다르다.

검증하기

검증은 작업 헌장의 완료 조건과 평가 루브릭을 기준으로 한다.

완료 조건별로 충족 여부를 표로 정리해줘.
증거가 없는 항목은 "미확인"으로 표시하고, 추측으로 통과 처리하지 마.

기록하기

마지막에 세션 장부를 남긴다.

SESSION_LOG 항목을 작성해줘.
포함할 것:
- 목표
- 읽은 문서
- 중요한 결정
- 검증 결과
- 남은 위험
- 다음 세션에 전달할 맥락
- 하네스 문서에 반영할 개선점

기록은 귀찮지만 자연어 하네스를 팀 자산으로 만드는 핵심이다.

10. 평가 루브릭과 회귀 체크

AI 결과물을 신뢰하려면 평가 기준이 필요하다. 자연어 하네스에서 평가 루브릭은 자동 채점기가 아니라 팀이 결과물을 보는 공통 렌즈다.

평가 루브릭 기본형

# EVAL_RUBRIC

## 공통 기준

- 목표 적합성: 작업 헌장의 목표를 충족하는가?
- 범위 준수: 비목표와 금지 범위를 침범하지 않았는가?
- 프로젝트 일관성: 기존 아키텍처, 용어, 문체와 맞는가?
- 검증 가능성: 결과가 증거와 함께 제시되는가?
- 운영 위험: 배포, 보안, 개인정보, 장애 대응 관점의 위험이 관리되는가?

## 판정

- Pass: 바로 수용 가능
- Needs Review: 사람이 특정 항목을 확인해야 함
- Revise: 재작업 필요
- Reject: 방향이 틀렸거나 위험함

판정 단계가 있어야 AI 결과가 “괜찮아 보임”에서 멈추지 않는다.

작업 유형별 루브릭

공통 루브릭만으로는 부족하다. 작업 유형별로 기준을 추가한다.

  • 기능 설계: 사용자 흐름, 예외 케이스, 데이터 계약, 마이그레이션 위험
  • 버그 수정: 재현 조건, 원인과 증상의 구분, 회귀 방지
  • 문서 작성: 독자, 최신성, 근거, 용어 일관성
  • 코드 리뷰: 정확성, 변경 범위, 테스트, 보안, 유지보수성
  • 장애 분석: 타임라인, 영향 범위, 근본 원인, 후속 조치

팀은 자주 하는 작업부터 루브릭을 만든다. 모든 것을 한 번에 만들 필요는 없다.

작업 유형별 상세 루브릭 예시

다음 예시는 그대로 복사해 팀에 맞게 줄여 쓸 수 있다.

## 버그 수정 루브릭

- 재현성: 버그가 발생하는 조건이 분명히 설명되었는가?
- 원인 구분: 근본 원인과 관찰된 증상을 구분했는가?
- 범위 제한: 수정이 문제 영역 밖으로 번지지 않았는가?
- 회귀 방지: 과거 유사 실패 항목을 확인했는가?
- 사용자 영향: 수정 전후 사용자 경험이 설명되었는가?
- 검증 증거: 재현 케이스, 테스트, 수동 확인 중 하나 이상의 증거가 있는가?

판정:
- Pass: 원인, 수정, 검증 증거가 모두 있음
- Needs Review: 검증은 있으나 원인 설명이 불완전함
- Revise: 수정은 했지만 재현/검증이 없음
- Reject: 증상만 가렸거나 범위를 과도하게 넓힘

## 기능 명세 루브릭

- 사용자 정의: 주요 사용자와 제외 사용자가 구분되었는가?
- 완료 조건: "무엇을 만든다"가 아니라 "어떤 상태가 된다"로 쓰였는가?
- 비목표: 이번 범위에서 하지 않을 일이 명확한가?
- 데이터 계약: API, 이벤트, 권한, 저장 데이터의 변화가 정리되었는가?
- 예외 흐름: 실패, 권한 없음, 빈 상태, 지연 상태가 포함되었는가?
- 검증 계획: 출시 전 확인할 항목이 역할별로 나뉘었는가?

## 코드 리뷰 루브릭

- 의도 일치: 변경이 PR 설명의 목적과 맞는가?
- 변경 범위: 필요 이상으로 넓은 파일 변경이 없는가?
- 기존 패턴: 프로젝트의 컴포넌트, API, 테스트 스타일을 따르는가?
- 위험 구역: 인증, 결제, 권한, 개인정보, 배포 설정 변경이 식별되었는가?
- 테스트 적합성: 변경 위험에 비례하는 테스트 또는 검증이 있는가?
- 리뷰 가능성: 사람이 확인해야 할 판단과 기계적으로 확인 가능한 항목이 분리되었는가?

루브릭은 완벽한 평가자가 아니다. 하지만 AI가 결과물을 낼 때 무엇을 증명해야 하는지 알려주는 계약이다.

소유권

하네스 지식베이스에도 소유자가 필요하다.

  • 프로젝트 오너: 비즈니스 목표와 범위를 승인한다.
  • 기술 오너: 아키텍처와 코드 제약을 관리한다.
  • 품질 오너: 평가 루브릭과 회귀 체크를 관리한다.
  • 문서 오너: 최신성, 구조, 링크를 관리한다.

작은 팀은 한 사람이 여러 역할을 맡아도 된다. 중요한 것은 누가 어떤 문서를 바꿀 수 있는지 정하는 것이다.

RACI로 보는 운영 책임

팀 규모가 커지면 “모두가 관리한다”는 말은 “아무도 관리하지 않는다”가 된다. 간단한 RACI 표를 둔다.

이 표는 관료주의를 만들기 위한 것이 아니다. AI가 참조하는 지식의 품질을 누가 보증하는지 분명히 하기 위한 최소 장치다.

월간 하네스 리뷰

주간 리뷰가 세션 단위라면, 월간 리뷰는 구조 단위다.

  • 이번 달 AI 세션에서 가장 많이 읽힌 문서는 무엇인가?
  • 한 번도 읽히지 않았지만 계속 남아 있는 문서는 무엇인가?
  • 반복 질문 상위 5개는 무엇인가?
  • 회귀 체크가 실제로 실패를 잡아낸 사례가 있는가?
  • 프롬프트 라이브러리에서 삭제할 항목은 무엇인가?
  • 프로젝트 구조 변경이 하네스 문서에 반영되었는가?

월간 리뷰의 목적은 하네스를 키우는 것이 아니라 정리하는 것이다. 자연어 하네스는 시간이 지날수록 더 길어지기 쉽다. 좋은 운영은 필요한 규칙을 추가하는 만큼 낡은 규칙을 제거한다.

12. 프로젝트별 하네스 운영 절차

팀 공용 원칙이 있어도 프로젝트마다 하네스는 달라야 한다. 결제 시스템, 관리자 도구, 마케팅 페이지, 데이터 파이프라인은 AI에게 제공해야 할 맥락과 위험 기준이 다르다.

프로젝트 시작 시 만들 문서

새 프로젝트를 시작할 때는 최소 다섯 문서를 만든다.

PROJECT_CONTEXT.md
GLOSSARY.md
AI_WORKFLOW.md
EVAL_RUBRIC.md
REGRESSION_NOTES.md

처음부터 완벽할 필요는 없다. 각 문서는 한 페이지 이하로 시작하고, 실제 세션에서 부족한 부분을 채운다.

작업 접수 절차

AI에게 맡길 작업은 바로 세션으로 보내지 않는다. 먼저 작업 헌장을 만든다.

  1. 사람이 작업 요청을 적는다.
  2. AI가 인터뷰 질문을 만든다.
  3. 사람이 중요한 질문에 답한다.
  4. AI가 작업 헌장 초안을 만든다.
  5. 사람이 범위와 완료 조건을 승인한다.
  6. 그다음 실행 세션으로 넘어간다.

작은 작업은 이 절차를 5분 안에 끝낼 수 있다. 큰 작업은 이 절차가 없으면 며칠을 잃을 수 있다.

세션 종료 절차

모든 의미 있는 AI 세션은 종료 기록을 남긴다.

세션 종료 체크:

- 작업 목표를 달성했는가?
- 완료 조건별 증거가 있는가?
- 사람이 확인해야 할 항목은 무엇인가?
- 새로 발견한 프로젝트 규칙은 무엇인가?
- 하네스 문서에 반영할 변경은 무엇인가?

세션 로그가 쌓이면 팀은 어떤 프롬프트가 효과적인지, 어떤 작업은 AI에게 맡기면 위험한지 알게 된다.

하네스 품질 지표

자연어 하네스도 성과를 볼 수 있다.

  • 작업 시작 전 확인 질문 수가 줄어드는가?
  • 같은 종류의 실패가 줄어드는가?
  • PR 리뷰에서 AI 결과물의 수정 요청이 줄어드는가?
  • 세션마다 반복 설명하는 프로젝트 맥락이 줄어드는가?
  • 신규 팀원이 AI 세션을 시작하는 시간이 줄어드는가?

정량 지표를 과하게 만들 필요는 없다. 중요한 것은 하네스가 실제 개발 흐름을 가볍게 만들고 있는지 보는 것이다.

도구별 운영 차이

자연어 하네스는 도구 중립적이어야 하지만, 세션 운영에서는 차이를 고려한다.

도구 유형강점주의점하네스 대응
Claude Code 계열긴 대화와 파일 작업에 강함세션이 길어지며 초기 제약이 흐려질 수 있음중간 압축과 세션 로그를 강제
Codex 계열코드베이스 탐색, 패치, 검증 루프에 강함명령 실행과 파일 변경 권한 관리가 중요권한 규칙과 검증 기준을 명확히 제공
IDE Chat 계열짧은 수정과 설명에 빠름프로젝트 전체 맥락 유지가 약할 수 있음작은 컨텍스트 패킷과 작업 헌장 사용
웹 챗 계열문서 초안, 사고 정리에 편함로컬 파일과 실제 검증 연결이 약함결과를 세션 로그와 검증 요청으로 분리

팀은 도구별 프롬프트를 완전히 따로 만들 필요는 없다. 다만 AI_WORKFLOW.md 안에 “이 도구를 쓸 때 특히 조심할 점”을 둔다.

작업 투입 프롬프트

이번 작업의 작업 헌장은 다음 파일이다:
- ai-harness/tasks/2026-06-auth-email-verification.md

진행 방식:
1. 작업 헌장을 읽고 완료 조건을 요약한다.
2. 컨텍스트 패킷에 포함된 문서를 읽는다.
3. 모호한 점이 있으면 질문한다.
4. 질문이 없으면 5단계 이내 계획을 제시한다.
5. 수행 후 평가 루브릭으로 자체 점검한다.
6. 마지막에 SESSION_LOG 초안을 작성한다.

이 프롬프트는 도구 중립적이다. Claude에서도 Codex에서도 쓸 수 있다.

세션 모드

모든 세션을 같은 방식으로 운영하면 비효율적이다. 작업 유형에 따라 모드를 선언한다.

이번 세션 모드: Read-only Review

규칙:
- 파일을 수정하지 않는다.
- 먼저 관련 문서를 읽고 위험과 누락을 찾는다.
- 최종 응답은 Findings, Questions, Suggested Changes로 나눈다.

이번 세션 모드: Assisted Implementation

규칙:
- 작업 헌장과 컨텍스트 패킷을 먼저 읽는다.
- 변경 계획을 제시한 뒤 진행한다.
- 변경 후 완료 조건별 검증 결과를 남긴다.

이번 세션 모드: Harness Maintenance

규칙:
- 제품 코드는 건드리지 않는다.
- 세션 로그와 회귀 노트를 검토한다.
- 하네스 문서 변경 제안만 작성한다.

모드를 선언하면 에이전트가 “무엇을 해도 되는지”를 더 잘 이해한다.

장기 세션에서의 압축 규칙

세션이 길어지면 중요한 맥락이 희미해진다. 중간에 요약을 요청한다.

지금까지의 세션을 다음 형식으로 압축해줘.

- 원래 목표:
- 확정된 결정:
- 변경된 범위:
- 이미 검증한 것:
- 아직 검증하지 못한 것:
- 다음 작업자가 반드시 알아야 할 것:
- 하네스 문서에 반영할 것:

이 요약은 다음 세션의 입력이 된다. 압축을 모델의 자동 기능에만 맡기지 말고, 팀이 필요한 항목을 지정한다.

권한과 위험

로컬 에이전트가 파일을 수정하거나 명령을 실행할 수 있다면, 자연어 하네스에 권한 규칙을 둔다.

권한 규칙:

- 사용자 데이터 삭제, 마이그레이션, 배포 명령은 승인 전 실행 금지.
- 보안 설정, 인증 정책, 결제 로직 변경은 먼저 계획과 위험을 보고.
- 대량 파일 변경은 변경 대상 목록을 먼저 제시.
- 외부 네트워크 접근이 필요하면 목적과 대상 URL을 설명.

권한은 “조심해줘”가 아니라 구체적인 승인 조건으로 써야 한다.

14. 하네스를 디벨롭하는 방법

자연어 하네스는 한 번 만들고 끝나는 문서가 아니다. 프로젝트가 변하고 팀의 AI 사용 방식이 성숙하면 하네스도 변해야 한다. 중요한 것은 변경을 우발적으로 하지 않는 것이다.

하네스 변경의 입력

하네스는 다음 신호를 바탕으로 바꾼다.

  • 같은 질문을 여러 세션에서 반복한다.
  • 같은 실수가 두 번 이상 재발한다.
  • 팀원이 프롬프트 라이브러리를 쓰지 않는다.
  • 평가 루브릭이 너무 추상적이라 판정이 흔들린다.
  • 프로젝트 구조가 바뀌었는데 컨텍스트 문서가 따라오지 못한다.
  • AI가 자꾸 범위를 넓히거나 엉뚱한 문서를 참조한다.

이런 신호는 하네스의 버그 리포트다.

하네스 버그 분류

하네스 문제를 그냥 “문서가 부족하다”로 보면 해결하기 어렵다. 다음처럼 분류한다.

유형증상대응
Context GapAI가 같은 프로젝트 맥락을 반복 질문PROJECT_CONTEXT.md 또는 컨텍스트 패킷 갱신
Ambiguity Gap핵심 용어를 다르게 해석GLOSSARY.md와 작업 헌장 템플릿 보강
Boundary GapAI가 범위를 넓힘비목표, 금지 범위, 권한 규칙 강화
Evaluation Gap결과가 좋아 보이지만 판정 어려움EVAL_RUBRIC.md 상세화
Regression Gap같은 실패가 반복REGRESSION_NOTES.md 승격
Adoption Gap팀원이 하네스를 쓰지 않음템플릿 축소, 시작 절차 단순화

분류가 있으면 하네스 개선이 감정적 회고에서 운영 작업으로 바뀐다.

하네스 변경 절차

변경 대상:
문제:

되돌릴 조건:

하네스 문서도 운영 시스템이다. 변경 이유가 없으면 나중에 왜 그런 규칙이 생겼는지 알 수 없다.

개선 실험 설계

하네스 변경은 작은 실험으로 다룬다.

예를 들어 “AI가 인증 작업에서 만료 세션과 권한 없음 상태를 혼동한다”는 문제가 있다면, REGRESSION_NOTES.md에 체크를 추가하고 2주 동안 인증 관련 세션에서 그 체크가 실제로 사용되는지 본다.

하네스 개선은 문서 작성량으로 평가하지 않는다. 다음 세션에서 실제 행동이 좋아졌는지로 평가한다.

좋은 하네스 개선의 예

나쁜 개선은 문서를 길게 만드는 것이다. 좋은 개선은 다음 중 하나를 달성한다.

  • 반복 설명을 줄인다.
  • 질문의 질을 높인다.
  • 위험한 작업을 더 빨리 드러낸다.
  • 검증을 더 구체적으로 만든다.
  • 오래된 정보를 더 쉽게 제거한다.
  • 새 팀원이 더 빨리 같은 품질의 세션을 시작하게 한다.

문서가 길어졌는데 이 중 아무것도 좋아지지 않았다면 개선이 아닐 수 있다.

자동화로 넘어가는 기준

자연어 하네스의 일부는 나중에 자동화할 수 있다. 다만 다음 조건이 충족될 때가 좋다.

  • 같은 절차가 최소 5회 이상 반복되었다.
  • 사람이 수동으로 해도 가치가 있다는 것이 확인되었다.
  • 입력과 출력 형식이 안정되었다.
  • 실패했을 때 사람이 쉽게 개입할 수 있다.
  • 자동화가 팀의 이해를 가리지 않는다.

먼저 문서로 운영하고, 그다음 반복되는 부분을 도구화한다. 이 순서가 팀에 더 안전하다.

자동화 후보 예시

자연어 하네스가 안정되면 다음을 자동화할 수 있다.

  • 새 작업 생성 시 작업 헌장 템플릿 자동 생성
  • 세션 로그에서 하네스 문서에 반영할 개선점만 모아 주간 리뷰 안건 생성
  • deprecated 문서가 컨텍스트 패킷에 들어가면 경고
  • 평가 결과에서 Unverified 항목이 있으면 PR 체크에 표시
  • REGRESSION_NOTES.md의 관련 항목을 작업 유형별로 추천

하지만 자동화는 하네스를 대체하지 않는다. 자동화는 팀이 이미 이해하고 합의한 절차를 덜 귀찮게 만드는 단계다.

15. 도입 로드맵과 체크리스트

자연어 하네스는 작은 범위에서 시작해야 한다. 처음부터 전사 표준을 만들려고 하면 문서만 커지고 실제 사용은 줄어든다.

1주차: 최소 하네스 만들기

  • 프로젝트 하나를 고른다.
  • PROJECT_CONTEXT.md 한 페이지를 만든다.
  • AI_WORKFLOW.md에 기본 실행 루프를 적는다.
  • EVAL_RUBRIC.md에 공통 평가 기준 5개를 적는다.
  • 작은 작업 하나를 작업 헌장으로 실행한다.
  • 세션 로그를 남긴다.

목표는 완성도가 아니라 반복 가능한 첫 흐름이다.

2주차: 실패 기록 추가

  • 지난 AI 세션의 실패 3개를 고른다.
  • 각 실패를 REGRESSION_NOTES.md에 기록한다.
  • 평가 루브릭에 회귀 체크 항목을 연결한다.
  • 프롬프트 라이브러리에 성공한 프롬프트 2개만 등록한다.

하네스는 성공 사례보다 실패 사례에서 빨리 자란다.

3주차: 팀 리뷰 시작

  • 주 1회 30분 리뷰를 잡는다.
  • 잘 된 세션 1개와 실패한 세션 1개를 본다.
  • 하네스 문서 변경 제안을 만든다.
  • 오래된 규칙을 삭제한다.

삭제는 중요하다. 자연어 하네스는 방치하면 길고 무거운 문서가 된다.

4주차: 프로젝트 템플릿화

  • 최소 구조를 템플릿으로 만든다.
  • 새 프로젝트에 복사할 수 있게 정리한다.
  • 프로젝트별로 반드시 달라져야 하는 항목을 표시한다.
  • 팀 온보딩 문서에 하네스 사용법을 추가한다.

이 시점에서 팀은 개인 프롬프트 모음에서 프로젝트 운영 체계로 넘어간다.

최종 체크리스트

도입 체크리스트:

- [ ] 프로젝트 컨텍스트가 한 페이지 안에 요약되어 있다.
- [ ] AI 세션 시작 절차가 문서화되어 있다.
- [ ] 작업 헌장 템플릿이 있다.
- [ ] 평가 루브릭이 있다.
- [ ] 회귀 노트가 있다.
- [ ] 세션 로그가 저장된다.
- [ ] 하네스 변경 이력이 남는다.
- [ ] 팀 리뷰 리듬이 있다.
- [ ] 오래된 문서를 제거하는 규칙이 있다.

자연어 하네스의 성공은 “AI가 한 번에 맞혔다”가 아니다. 팀이 더 적은 반복 설명으로, 더 안정적인 결과를, 더 잘 검증할 수 있게 되는 것이다.

16. 첫 번째 ai-harness 만들기

이 장은 자연어 하네스를 실제 프로젝트에 처음 설치하는 과정을 다룬다. 여기서 설치란 도구를 깔거나 코드를 작성하는 일이 아니다. 팀이 Claude나 Codex에게 반복해서 제공할 작업 환경을 문서로 만드는 일이다.

목표는 하루 안에 최소 하네스를 만드는 것이다. 완벽한 지식베이스가 아니라, 첫 AI 세션을 이전보다 덜 위험하게 시작할 수 있는 구조를 만든다.

0단계: 적용할 프로젝트 고르기

처음부터 가장 복잡한 핵심 시스템을 고르지 않는다. 다음 조건을 만족하는 프로젝트가 좋다.

  • 팀이 자주 수정한다.
  • AI를 이미 가끔 사용하고 있다.
  • 반복되는 설명이나 실수가 있다.
  • 프로젝트 맥락을 아는 사람이 옆에 있다.
  • 실패해도 작은 작업부터 실험할 수 있다.

예를 들어 관리자 콘솔, 내부 운영 도구, 문서화가 부족한 프런트엔드 앱, 반복 버그가 많은 백엔드 서비스가 좋은 후보가 될 수 있다.

1단계: 폴더 만들기

프로젝트 루트나 문서 저장소에 다음 구조를 만든다.

ai-harness/
  README.md
  PROJECT_CONTEXT.md
  GLOSSARY.md
  AI_WORKFLOW.md
  EVAL_RUBRIC.md
  REGRESSION_NOTES.md
  PROMPT_LIBRARY.md
  HARNESS_CHANGELOG.md
  tasks/
  sessions/
  decisions/
  workflows/

2단계: README 작성

- 관리자 콘솔 진입 라우팅
- 세션 상태 확인 흐름
- 권한 가드
- 에러/리다이렉트 처리

- 수정 작업이 필요하면 후속 작업 헌장 초안이 작성된다.

18. 사례 연구: 신규 기능 명세 세션

버그 분석보다 더 어려운 작업은 신규 기능 명세다. 버그는 적어도 잘못된 상태가 보이지만, 신규 기능은 무엇을 만들지부터 흔들린다. 자연어 하네스는 신규 기능을 바로 구현하지 않고, 팀이 합의할 수 있는 명세로 좁힌다.

시작 요청

관리자가 멤버 초대할 때 권한을 미리 지정할 수 있게 해줘.

이 요청은 단순해 보이지만 중요한 질문이 숨어 있다.

  • 권한은 어떤 종류가 있는가?
  • 초대받은 사용자가 가입하지 않으면 권한은 어디에 저장되는가?
  • 기존 초대 링크와 충돌하지 않는가?
  • 권한 변경 감사 로그는 언제 남는가?
  • 잘못 초대했을 때 취소나 수정이 가능한가?
  • 초대 수락 전에 권한 정책이 바뀌면 어떻게 되는가?

Wonder 단계

먼저 문제 공간을 넓힌다.

너는 제품 명세 인터뷰어다.
아직 구현 방식을 제안하지 말고, "관리자가 멤버 초대 시 권한을 미리 지정"한다는 요청의 목적과 대안을 탐색해줘.

Wonder 질문:
- 이 기능이 해결하려는 실제 문제는 무엇인가?
- 현재 초대 후 권한 변경 흐름에서 어떤 비용이 발생하는가?
- 초대 시 권한을 지정하지 않고도 문제를 해결할 방법은 있는가?
- 잘못된 권한 초대가 발생하면 어떤 위험이 있는가?

이 단계의 목표는 구현 아이디어를 확정하는 것이 아니라, 기능의 존재 이유를 분명히 하는 것이다.

Ontology 단계

다음으로 용어를 좁힌다.

Ontology 질문:
- "관리자"는 workspace admin인가 organization owner인가?
- "멤버"는 신규 가입자, 기존 사용자, 외부 게스트를 모두 포함하는가?
- "권한"은 role인가 permission set인가?
- "미리 지정"은 초대 생성 시점, 초대 수락 시점, 계정 생성 시점 중 언제 적용되는가?
- 초대가 만료되면 사전 지정 권한도 함께 폐기되는가?

이 질문에 답하지 않으면 AI는 익숙한 권한 모델을 마음대로 가져온다. 그것이 프로젝트의 실제 권한 체계와 다르면 위험하다.

Seed Spec 만들기

Ouroboros의 명세 우선 흐름을 자연어로 옮기면 다음과 같은 Seed Spec이 된다.

# Seed Spec: 초대 시 권한 사전 지정

## 문제

workspace admin은 신규 멤버를 초대한 뒤 별도 화면에서 권한을 조정해야 한다.
이 과정에서 멤버가 잘못된 기본 권한으로 잠시 접근하거나, 관리자가 권한 변경을 잊는 문제가 발생한다.

## 목표

workspace admin이 초대 생성 시 초대받을 멤버의 초기 role을 지정할 수 있게 한다.
초대 수락 후 멤버는 지정된 role로 workspace에 합류한다.

## 비목표

- role 체계를 재설계하지 않는다.
- organization owner 권한 정책을 변경하지 않는다.
- 초대 링크 보안 정책을 변경하지 않는다.
- 대량 초대 기능은 이번 범위에 포함하지 않는다.

## 핵심 용어

- inviter: 초대를 생성하는 workspace admin
- invitee: 초대를 수락할 사용자
- initial role: 초대 생성 시 지정되고, 수락 시 적용되는 role
- pending invite: 아직 수락되지 않은 초대

## 완료 조건

- 초대 생성 시 role을 선택할 수 있다.
- 선택 가능한 role은 inviter의 권한보다 높을 수 없다.
- 초대 수락 시 initial role이 적용된다.
- 초대 취소 시 pending role 정보도 폐기된다.
- 감사 로그에는 초대 생성자, invitee, initial role이 남는다.

## 남은 질문

- 초대 수락 전에 role 정책이 변경되면 pending invite를 무효화할 것인가?
- 기존 pending invite의 role을 수정할 수 있어야 하는가?

Seed Spec은 구현 지시서가 아니다. 팀이 논의할 수 있는 최소 명세다.

- 결제 plan에 따라 초대 가능한 role이 달라질 수 있다.

설계 옵션 만들기

명세가 어느 정도 선명해지면 AI에게 옵션을 만들게 한다.

Seed Spec을 기준으로 설계 옵션을 3개 제시해줘.
각 옵션마다 다음을 포함해줘:
- 사용자 흐름
- 데이터 계약 변화
- 권한/감사 로그 영향
- 장점
- 위험
- 먼저 검증해야 할 질문

좋은 응답은 옵션을 단순 UI 차이가 아니라 운영 위험 차이로 나눈다.

옵션 A: 초대 생성 시 role을 필수 선택
- 장점: 누락이 없다.
- 위험: 단순 초대 흐름이 복잡해진다.

옵션 B: 기본 role을 유지하되 고급 설정으로 role 선택
- 장점: 기존 흐름 영향이 작다.
- 위험: 관리자가 여전히 설정을 놓칠 수 있다.

옵션 C: 초대 템플릿을 만들어 role과 메시지를 함께 저장
- 장점: 반복 초대에 유리하다.
- 위험: 이번 범위를 크게 넘는다.

평가와 결정

팀은 평가 루브릭으로 옵션을 고른다.

판정:
옵션 B를 1차 범위로 선택한다.

이유:
- 기존 초대 흐름의 마찰을 최소화한다.
- 권한 지정 기능을 제공하되 필수 입력으로 만들지 않는다.
- 대량 초대나 템플릿 기능으로 범위가 커지는 것을 막는다.

후속 확인:
- 기본 role이 무엇인지 정책 확인 필요
- 초대 수락 전 role 수정 가능 여부는 별도 작업으로 분리

좋은 명세 세션은 모든 결정을 끝내지 않는다. 어떤 결정을 이번에 하고, 어떤 결정을 다음 작업으로 넘길지 분리한다.

19. 팀 거버넌스와 변경 관리

자연어 하네스는 팀 공용 지식베이스다. 따라서 개인 노트처럼 관리하면 오래가지 않는다. 누가 바꿀 수 있는지, 어떤 변경은 리뷰가 필요한지, 언제 삭제해야 하는지 정해야 한다.

거버넌스의 목적은 문서를 무겁게 만드는 것이 아니다. AI가 참조하는 지식이 팀의 실제 합의와 계속 일치하도록 만드는 것이다.

문서 등급

모든 문서를 같은 무게로 다루지 않는다.

등급예시변경 규칙
CorePROJECT_CONTEXT, EVAL_RUBRIC, REGRESSION_NOTES리뷰 필수
WorkflowAI_WORKFLOW, workflows/*관련 사용자 1명 이상 확인
Recordsessions/, tasks/작성자 책임으로 추가 가능
LibraryPROMPT_LIBRARY, examples/*성공 사례 링크 필요
Deprecatedold docs읽지 말기 사유 표시

Core 문서는 AI 행동에 직접 영향을 준다. 따라서 변경이 작아 보여도 리뷰한다.

변경 승인 기준

하네스 문서 변경은 다음 기준으로 승인한다.

  • 실제 세션, 이슈, 리뷰, 장애, 사용자 피드백 중 하나 이상의 근거가 있는가?
  • AI의 다음 행동이 어떻게 달라져야 하는지 설명하는가?
  • 특정 프로젝트에만 적용되는 규칙을 전역 규칙처럼 쓰지 않았는가?
  • 기존 규칙과 충돌하지 않는가?
  • 문서가 길어졌다면 검색성과 읽기성이 유지되는가?

이 기준은 특히 프롬프트 라이브러리에 중요하다. “좋아 보이는 프롬프트”가 아니라 “검증된 상황에서 효과가 있었던 프롬프트”만 남긴다.

하네스 변경 PR 예시

제목:
인증 상태 구분 회귀 체크 추가

변경 이유:
최근 두 번의 AI 세션에서 만료 세션과 권한 없음 상태를 혼동했다.

근거:
- sessions/2026-05-28-role-guard.md
- sessions/2026-06-06-auth-empty-screen.md

변경 내용:

- REGRESSION_NOTES.md에 인증 상태 구분 체크 추가
- EVAL_RUBRIC.md의 버그 수정 루브릭에 상태 구분 항목 추가

기대하는 AI 행동 변화:
인증 관련 작업에서 세 상태를 하나의 "로그인 문제"로 뭉뚱그리지 않고 별도 완료 조건으로 검증한다.

되돌릴 기준:
인증 정책 변경으로 세 상태 정의가 더 이상 맞지 않게 되면 폐기한다.

좋은 변경 PR은 하네스 문서를 바꾼 사실보다, AI 세션의 어떤 실패를 줄일지 설명한다.

20. 성숙도 모델과 다음 단계

자연어 하네스는 한 번에 완성되지 않는다. 팀은 보통 개인 프롬프트에서 시작해, 프로젝트 지식베이스, 팀 운영 절차, 부분 자동화로 발전한다. 이 장은 현재 위치를 진단하고 다음 단계를 고르는 기준을 제공한다.

Level 0: 개인 프롬프트

특징:

  • 팀원마다 각자 프롬프트를 쓴다.
  • 프로젝트 맥락은 매번 대화로 설명한다.
  • 결과 검증은 개인 경험에 의존한다.
  • 좋은 프롬프트가 있어도 공유되지 않는다.

대표 증상:

  • “이거 전에 설명했는데”가 반복된다.
  • AI 결과물의 품질이 사용자마다 크게 다르다.
  • 실패 원인이 저장되지 않는다.

다음 단계:

  • PROJECT_CONTEXT.md 한 페이지를 만든다.
  • 공통 세션 시작 프롬프트를 만든다.

Level 1: 최소 프로젝트 하네스

특징:

  • ai-harness/ 폴더가 있다.
  • 프로젝트 컨텍스트와 평가 루브릭이 있다.
  • 작업 헌장과 세션 로그를 일부 사용한다.

대표 증상:

  • 문서는 있지만 최신성이 흔들린다.
  • 팀원 일부만 사용한다.
  • 템플릿이 너무 길거나 추상적이다.

다음 단계:

  • 주간 하네스 리뷰를 시작한다.
  • 회귀 노트를 실제 실패 사례에서 만들기 시작한다.

Level 2: 팀 운영 하네스

특징:

  • 작업 유형별 워크플로우가 있다.
  • 평가 루브릭과 회귀 체크가 실제 리뷰에 쓰인다.
  • 하네스 문서 변경에 소유자와 리뷰 절차가 있다.
  • Claude, Codex, IDE Chat 등 도구별 세션 규칙이 있다.

대표 증상:

  • 신규 팀원이 AI 세션을 더 빨리 시작한다.
  • 반복 질문이 줄어든다.
  • AI가 만든 결과물의 리뷰 코멘트가 더 구체적이다.

다음 단계:

  • 세션 로그에서 반복 개선점을 추출한다.
  • 안정된 절차를 일부 자동화한다.

Level 3: 관찰 가능한 하네스

특징:

  • 세션 로그, 평가 결과, 회귀 체크가 축적된다.
  • 어떤 하네스 변경이 AI 행동을 개선했는지 추적한다.
  • 실패 유형을 Context Gap, Ambiguity Gap, Evaluation Gap 등으로 분류한다.
  • 오래된 문서를 폐기하는 절차가 작동한다.

대표 증상:

  • 하네스 변경이 감이 아니라 근거로 이루어진다.
  • 프롬프트 라이브러리에 검증 이력이 붙는다.
  • 프로젝트별 하네스를 팀 템플릿으로 역수출할 수 있다.

다음 단계:

  • 반복 절차를 도구화한다.
  • 문서 최신성, 회귀 체크, 평가 결과를 자동 점검한다.

Level 4: 부분 자동화 하네스

특징:

  • 작업 헌장 생성, 세션 로그 수집, 회귀 체크 추천이 자동화된다.
  • PR이나 이슈에 하네스 체크리스트가 연결된다.
  • AI 세션 결과가 지식베이스 개선 제안으로 이어진다.

주의:

자동화는 성숙한 자연어 절차 위에 올려야 한다. 합의되지 않은 절차를 자동화하면, 팀은 왜 그런 규칙이 있는지 모른 채 도구에 끌려간다.

자가 진단 체크

우리 팀의 현재 수준:

- [ ] AI 세션 시작 문서가 있다.
- [ ] 프로젝트 컨텍스트가 최신 상태다.
- [ ] 작업 헌장 없이 위험한 작업을 시작하지 않는다.
- [ ] 평가 루브릭으로 결과를 판정한다.
- [ ] 회귀 노트가 실제 실패에서 만들어진다.
- [ ] 하네스 문서 변경 이력이 있다.
- [ ] 팀원이 하네스 리뷰를 정기적으로 한다.
- [ ] 오래된 문서가 deprecated 처리된다.
- [ ] 안정된 절차 일부가 자동화되어 있다.

체크 수가 적다고 나쁜 팀은 아니다. 중요한 것은 다음 한 단계를 고르는 것이다.

21. 상태 장부 기반 집필과 자체 검증 루프

자연어 하네스 책을 만드는 과정도 하나의 하네스 작업이다. 따라서 책을 작성하는 에이전트는 단순히 장을 쓰고 끝내면 안 된다. 현재 진행 상태를 _state/<topic>.md에 기록하고, 스스로 품질을 검토하고, 부족한 부분을 보강하고, 다시 빌드해야 한다.

이 장은 책 작성 자체를 Spec -> Draft -> Review -> Reinforce -> Verify -> Evolve 루프로 운영하는 방법을 정리한다.

왜 상태 장부가 필요한가

긴 문서 작업은 쉽게 방향을 잃는다. 특히 AI가 책을 작성할 때는 다음 문제가 자주 생긴다.

  • 초반 요구사항은 좋게 반영했지만 후반 장에서 희미해진다.
  • 목차는 충분해 보이지만 장별 내용이 얕다.
  • 템플릿은 많지만 완성 사례가 없다.
  • 출처는 달려 있지만 실제 주장과 느슨하게 연결된다.
  • 빌드는 통과하지만 학습 목표는 충족하지 못한다.
  • 사용자의 피드백을 반영했는지 나중에 추적하기 어렵다.

상태 장부는 이런 문제를 막기 위한 작업 메모리가 아니라, 에이전트가 스스로를 검증하는 운영 장부다.

상태 장부의 최소 구조

책 작성용 _state/<topic>.md에는 최소한 다음 섹션이 있어야 한다.

## 진행 스냅샷

- 현재 단계
- 완료된 장
- 남은 장
- 마지막 빌드 상태

## 사용자 요구사항 반영

- 사용자가 명시한 목표
- 대상 독자
- 반드시 반영할 출처
- 제외할 방향

## 자체 리뷰

- 분량 적정성
- 목차 완결성
- 실무 적용 가능성
- 사례 충분성
- 템플릿 재사용성
- 출처 연결성
- 빌드/검증 상태

## 품질 게이트

- 통과 기준
- 현재 판정
- 미달 항목

## 보강 반복 로그

- Iteration 1: 발견한 문제, 보강 내용, 검증 결과
- Iteration 2: 발견한 문제, 보강 내용, 검증 결과

중요한 것은 “완료했다”가 아니라 “왜 충분하다고 판단했는지”를 남기는 것이다.

자체 리뷰 루브릭

책 작성 후 에이전트는 다음 기준으로 자체 리뷰를 수행한다.

# 도서 자체 리뷰 루브릭

## 1. 의도 적합성

- 사용자가 요청한 대상 독자에게 맞는가?
- 사용자가 제외하라고 한 방향을 피했는가?
- 최종 목표가 책 전체에서 반복적으로 유지되는가?

## 2. 구조 완결성

- 개념, 절차, 사례, 운영, 발전 과정이 모두 있는가?
- 목차가 학습 순서를 만든는가?
- 부록이 본문에서 실제로 쓰이는가?

## 3. 깊이와 분량

- 장별 내용이 단순 요약에 머물지 않는가?
- 이전 유사 산출물과 비교해 과도하게 얕지 않은가?
- 핵심 장에는 예시, 체크리스트, 실패 모드가 있는가?

## 4. 실무 적용 가능성

- 독자가 따라 할 수 있는 절차가 있는가?
- 완성 샘플이 있는가?
- 팀 단위 운영 규칙이 있는가?
- 도입 후 유지보수 방법이 있는가?

## 5. 검증 가능성

- mdBook doctor가 통과하는가?
- mdBook build가 통과하는가?
- SUMMARY 링크가 깨지지 않는가?
- 각 장 마지막 H2가 `출처`인가?

## 6. 보강 필요 판정

- Pass: 그대로 완료 가능
- Reinforce: 일부 장 보강 필요
- Restructure: 목차 재구성 필요
- Re-scope: 사용자 목표 재확인 필요

이 루브릭은 사람이 읽는 리뷰에도 쓸 수 있고, AI가 자기 산출물을 점검할 때도 쓸 수 있다.

반복 보강 루프

도서 작성 루프는 다음 순서로 진행한다.

  1. Draft: 목차와 장 초안을 작성한다.
  2. Build: mdBook doctor와 build를 실행한다.
  3. Self Review: 자체 리뷰 루브릭으로 책을 평가한다.
  4. Gap List: 부족한 장, 빠진 사례, 얕은 템플릿을 목록화한다.
  5. State Update: _state/<topic>.md에 리뷰 결과와 보강 계획을 기록한다.
  6. Reinforce: 부족한 장을 보강하거나 새 장을 추가한다.
  7. Rebuild: 다시 doctor와 build를 실행한다.
  8. Decision: 품질 게이트를 통과하면 완료, 아니면 다음 반복으로 간다.

이 루프는 사용자가 명시적으로 멈추기 전까지 “충분한 수준”에 도달할 때까지 반복할 수 있어야 한다.

충분한 수준의 정의

“충분하다”는 말은 주관적이다. 그래서 책마다 충분함의 기준을 정해야 한다.

자연어 하네스 학습서의 경우 충분한 수준은 다음과 같다.

  • 독자가 왜 자연어 하네스가 필요한지 이해한다.
  • 독자가 ai-harness/ 최소 구조를 만들 수 있다.
  • 독자가 Claude나 Codex 세션을 하네스 문서와 연결해 실행할 수 있다.
  • 독자가 결과를 평가하고 세션 로그를 남길 수 있다.
  • 팀이 회귀 노트, 평가 루브릭, 변경 이력을 관리할 수 있다.
  • 책 자체도 _state를 통해 반복 보강 가능하다.

이 기준을 만족하지 못하면 책은 완료가 아니라 다음 보강 반복으로 넘어간다.

Ouroboros는 명세 우선 워크플로우와 에이전트 OS 관점을 제공한다. 특히 막연한 요청을 인터뷰, 명세, 실행, 평가, 진화의 루프로 바꾸는 발상이 이 책의 핵심 구조에 영향을 주었다.

  • 6장: 작업 헌장
  • 7장: Wonder와 Ontology 질문
  • 9장: 실행 루프
  • 12장: 프로젝트별 운영 절차
  • 14장: 하네스 디벨롭 과정

  • 18장: 신규 기능의 Seed Spec 흐름

Part 4 실전 사례의 역할

초기 초안은 개념과 템플릿 중심이라 실제 프로젝트에 도입하는 과정이 부족했다. Part 4는 세 원천의 아이디어를 한 프로젝트의 운영 흐름으로 묶는다.

  • Prompt Engineering Guide의 패턴은 실제 세션 프롬프트로 변환했다.
  • Awesome Harness Engineering의 컨텍스트, 메모리, 평가, human-in-the-loop 요소는 팀 문서와 리뷰 절차로 변환했다.
  • Ouroboros의 interview -> spec -> run -> evaluate -> evolve 흐름은 작업 헌장, 컨텍스트 패킷, 세션 로그, 하네스 변경 이력으로 변환했다.

Part 5 자체 검증 루프의 역할

Part 5는 이 책의 작성 과정 자체를 하네스 대상으로 삼는다. mdBook 산출물도 자연어 하네스의 원칙을 따라야 하므로, _state 장부, 자체 리뷰 루브릭, 품질 게이트, 반복 보강 로그를 통해 책이 충분한 수준에 도달할 때까지 개선되도록 설계했다.

C. 용어집

자연어 하네스

AI 에이전트가 프로젝트 안에서 일관되게 일하도록 돕는 문서 기반 실행 환경. 코드 런타임 대신 작업 헌장, 컨텍스트 패킷, 평가 루브릭, 세션 로그, 회귀 노트 같은 자연어 산출물로 구성한다.

프롬프트

AI에게 주는 입력 문장. 이 책에서는 단순 요청이 아니라 역할, 목표, 입력, 제약, 절차, 출력 형식을 담은 작업 인터페이스로 본다.

작업 헌장

이번 작업의 배경, 목표, 비목표, 범위, 완료 조건, 검증 방법을 정리한 문서. AI가 바로 실행하기 전에 사람과 팀의 의도를 고정하는 역할을 한다.

컨텍스트 패킷

AI 세션 시작 시 읽어야 할 문서 묶음. 전체 지식베이스가 아니라 이번 작업에 필요한 최소 충분 맥락을 제공한다.

평가 루브릭

AI 결과물을 팀 기준으로 판정하기 위한 평가 항목. 목표 적합성, 범위 준수, 프로젝트 일관성, 검증 가능성, 운영 위험 등을 포함한다.

회귀 체크

과거에 발생한 실패가 다시 생기지 않도록 확인하는 항목. 일반 베스트 프랙티스보다 프로젝트 고유 실패 경험을 우선한다.

세션 로그

AI 세션의 목표, 읽은 문서, 결정, 수행 결과, 검증, 남은 위험, 다음 세션에 전달할 맥락을 기록한 문서.

Wonder 질문

문제 공간을 넓히는 질문. 왜 필요한지, 누가 영향을 받는지, 다른 해결책은 있는지 탐색한다.

Ontology 질문

핵심 단어의 의미를 좁히는 질문. 사용자, 완료, 실패, 관리자 같은 용어가 팀 안에서 같은 뜻으로 쓰이게 만든다.

명세 우선

AI가 바로 결과물을 만들기 전에 문제, 용어, 범위, 완료 조건, 검증 기준을 먼저 정리하는 접근.

하네스 변경 이력

하네스 문서 자체의 변경 이유와 효과를 기록하는 장부. 자연어 하네스가 시간이 지나며 왜 바뀌었는지 보존한다.

Context Gap

AI가 작업에 필요한 프로젝트 맥락을 받지 못해 반복 질문을 하거나 잘못 추측하는 상태. 보통 PROJECT_CONTEXT.md나 컨텍스트 패킷을 보강해 해결한다.

Boundary Gap

AI가 작업 범위, 비목표, 금지된 변경을 충분히 이해하지 못해 일을 키우는 상태. 작업 헌장과 권한 규칙을 강화해야 한다.

Evaluation Gap

결과가 좋아 보이지만 팀 기준으로 통과인지 판단하기 어려운 상태. 평가 루브릭과 완료 조건을 더 구체화해야 한다.

Stale 문서

현재 프로젝트 상태와 맞지 않지만 지식베이스에 남아 있어 AI가 잘못된 근거로 사용할 수 있는 문서. deprecated 또는 archived 상태와 대체 문서를 표시해야 한다.

상태 장부

책, 프로젝트, AI 세션의 진행 상태와 판단 근거를 기록하는 문서. 이 vault에서는 _state/<topic>.md가 상태 장부 역할을 한다.

품질 게이트

산출물을 완료로 볼 수 있는 최소 기준. 문서 작업에서는 의도 적합성, 구조 완결성, 실무 적용 가능성, 출처, 빌드 검증 등을 포함한다.

보강 반복 로그

자체 리뷰나 사용자 피드백으로 발견한 부족한 점, 보강한 내용, 검증 결과를 반복 단위로 기록한 로그.
