21. 상태 장부 기반 집필과 자체 검증 루프

자연어 하네스 책을 만드는 과정도 하나의 하네스 작업이다. 따라서 책을 작성하는 에이전트는 단순히 장을 쓰고 끝내면 안 된다. 현재 진행 상태를 _state/<topic>.md에 기록하고, 스스로 품질을 검토하고, 부족한 부분을 보강하고, 다시 빌드해야 한다.

이 장은 책 작성 자체를 Spec -> Draft -> Review -> Reinforce -> Verify -> Evolve 루프로 운영하는 방법을 정리한다.

왜 상태 장부가 필요한가

긴 문서 작업은 쉽게 방향을 잃는다. 특히 AI가 책을 작성할 때는 다음 문제가 자주 생긴다.

  • 초반 요구사항은 좋게 반영했지만 후반 장에서 희미해진다.
  • 목차는 충분해 보이지만 장별 내용이 얕다.
  • 템플릿은 많지만 완성 사례가 없다.
  • 출처는 달려 있지만 실제 주장과 느슨하게 연결된다.
  • 빌드는 통과하지만 학습 목표는 충족하지 못한다.
  • 사용자의 피드백을 반영했는지 나중에 추적하기 어렵다.

상태 장부는 이런 문제를 막기 위한 작업 메모리가 아니라, 에이전트가 스스로를 검증하는 운영 장부다.

상태 장부의 최소 구조

책 작성용 _state/<topic>.md에는 최소한 다음 섹션이 있어야 한다.

## 진행 스냅샷

- 현재 단계
- 완료된 장
- 남은 장
- 마지막 빌드 상태

## 사용자 요구사항 반영

- 사용자가 명시한 목표
- 대상 독자
- 반드시 반영할 출처
- 제외할 방향

## 자체 리뷰

- 분량 적정성
- 목차 완결성
- 실무 적용 가능성
- 사례 충분성
- 템플릿 재사용성
- 출처 연결성
- 빌드/검증 상태

## 품질 게이트

- 통과 기준
- 현재 판정
- 미달 항목

## 보강 반복 로그

- Iteration 1: 발견한 문제, 보강 내용, 검증 결과
- Iteration 2: 발견한 문제, 보강 내용, 검증 결과

중요한 것은 “완료했다”가 아니라 “왜 충분하다고 판단했는지”를 남기는 것이다.

자체 리뷰 루브릭

책 작성 후 에이전트는 다음 기준으로 자체 리뷰를 수행한다.

# 도서 자체 리뷰 루브릭

## 1. 의도 적합성

- 사용자가 요청한 대상 독자에게 맞는가?
- 사용자가 제외하라고 한 방향을 피했는가?
- 최종 목표가 책 전체에서 반복적으로 유지되는가?

## 2. 구조 완결성

- 개념, 절차, 사례, 운영, 발전 과정이 모두 있는가?
- 목차가 학습 순서를 만든는가?
- 부록이 본문에서 실제로 쓰이는가?

## 3. 깊이와 분량

- 장별 내용이 단순 요약에 머물지 않는가?
- 이전 유사 산출물과 비교해 과도하게 얕지 않은가?
- 핵심 장에는 예시, 체크리스트, 실패 모드가 있는가?

## 4. 실무 적용 가능성

- 독자가 따라 할 수 있는 절차가 있는가?
- 완성 샘플이 있는가?
- 팀 단위 운영 규칙이 있는가?
- 도입 후 유지보수 방법이 있는가?

## 5. 검증 가능성

- mdBook doctor가 통과하는가?
- mdBook build가 통과하는가?
- SUMMARY 링크가 깨지지 않는가?
- 각 장 마지막 H2가 `출처`인가?

## 6. 보강 필요 판정

- Pass: 그대로 완료 가능
- Reinforce: 일부 장 보강 필요
- Restructure: 목차 재구성 필요
- Re-scope: 사용자 목표 재확인 필요

이 루브릭은 사람이 읽는 리뷰에도 쓸 수 있고, AI가 자기 산출물을 점검할 때도 쓸 수 있다.

반복 보강 루프

도서 작성 루프는 다음 순서로 진행한다.

  1. Draft: 목차와 장 초안을 작성한다.
  2. Build: mdBook doctor와 build를 실행한다.
  3. Self Review: 자체 리뷰 루브릭으로 책을 평가한다.
  4. Gap List: 부족한 장, 빠진 사례, 얕은 템플릿을 목록화한다.
  5. State Update: _state/<topic>.md에 리뷰 결과와 보강 계획을 기록한다.
  6. Reinforce: 부족한 장을 보강하거나 새 장을 추가한다.
  7. Rebuild: 다시 doctor와 build를 실행한다.
  8. Decision: 품질 게이트를 통과하면 완료, 아니면 다음 반복으로 간다.

이 루프는 사용자가 명시적으로 멈추기 전까지 “충분한 수준”에 도달할 때까지 반복할 수 있어야 한다.

품질 게이트 예시

다음은 자연어 하네스 학습서에 적용할 수 있는 품질 게이트다.

# 품질 게이트

필수 통과:
- [ ] 사용자의 핵심 목표가 00_overview와 최종 장에 모두 반영되어 있다.
- [ ] 주요 출처 3개가 소스 맵에 연결되어 있다.
- [ ] 개념 장뿐 아니라 실전 사례 장이 있다.
- [ ] 팀 단위 운영 절차가 있다.
- [ ] 상태 장부 기반 반복 보강 절차가 있다.
- [ ] 모든 SUMMARY 링크가 유효하다.
- [ ] 모든 장의 마지막 H2가 `출처`다.
- [ ] mdBook doctor가 통과한다.
- [ ] mdBook build가 통과한다.

권장 통과:
- [ ] 핵심 장은 템플릿만이 아니라 완성 예시를 포함한다.
- [ ] 독자가 바로 복사할 수 있는 운영 문서 예시가 있다.
- [ ] 실패 모드와 회귀 체크가 있다.
- [ ] 도입 후 성숙도 모델이나 다음 단계가 있다.

필수 항목이 하나라도 실패하면 완료로 표시하지 않는다. 권장 항목이 부족하면 Reinforce로 표시하고 보강한다.

_state에 남기는 반복 로그

보강 반복은 다음 형식으로 남긴다.

## 보강 반복 로그

### Iteration 1

- Trigger: 사용자 피드백 - "이전 코드 기반 책보다 내용이 줄어든 것 같다."
- Self Review: 장별 깊이와 실무 적용 사례 부족.
- Gap List:
  - 실전 적용 사례 없음
  - 팀 거버넌스 부족
  - 성숙도 모델 없음
- Actions:
  - 16-20장 추가
  - 08, 10, 11, 13, 14장 보강
- Verification:
  - doctor passed
  - build passed
- Decision: Pass with follow-up

### Iteration 2

- Trigger: 사용자 피드백 - "보강 방식도 _state에 업데이트되면서 처리되어야 한다."
- Self Review: 자체 검증 루프가 책과 템플릿에 명문화되지 않음.
- Gap List:
  - 상태 장부 기반 집필 절차 없음
  - 품질 게이트 없음
  - 반복 보강 로그 템플릿 없음
- Actions:
  - 21장 추가
  - _system/TEMPLATE.md 확장
- Verification:
  - pending
- Decision: pending

이 기록은 나중에 “왜 책이 이렇게 커졌는지”, “어떤 피드백이 어떤 변경으로 이어졌는지”를 설명한다.

충분한 수준의 정의

“충분하다”는 말은 주관적이다. 그래서 책마다 충분함의 기준을 정해야 한다.

자연어 하네스 학습서의 경우 충분한 수준은 다음과 같다.

  • 독자가 왜 자연어 하네스가 필요한지 이해한다.
  • 독자가 ai-harness/ 최소 구조를 만들 수 있다.
  • 독자가 Claude나 Codex 세션을 하네스 문서와 연결해 실행할 수 있다.
  • 독자가 결과를 평가하고 세션 로그를 남길 수 있다.
  • 팀이 회귀 노트, 평가 루브릭, 변경 이력을 관리할 수 있다.
  • 책 자체도 _state를 통해 반복 보강 가능하다.

이 기준을 만족하지 못하면 책은 완료가 아니라 다음 보강 반복으로 넘어간다.
