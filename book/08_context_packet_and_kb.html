자연어 하네스와 팀 프롬프팅

8. 컨텍스트 패킷과 프로젝트 지식베이스

컨텍스트 패킷은 AI 세션을 시작할 때 제공하는 최소 프로젝트 맥락이다. 지식베이스 전체를 던지는 것이 아니라, 이번 작업에 필요한 문서 묶음을 고르는 행위다.

자연어 하네스에서 컨텍스트 관리는 가장 현실적인 성능 개선 수단이다. 모델을 바꾸지 않아도, 입력 맥락의 품질을 높이면 결과가 크게 안정된다.

지식베이스 기본 구조

프로젝트 루트나 문서 저장소에 다음 구조를 둔다.

ai-harness/
  README.md
  PROJECT_CONTEXT.md
  GLOSSARY.md
  AI_WORKFLOW.md
  EVAL_RUBRIC.md
  REGRESSION_NOTES.md
  PROMPT_LIBRARY.md
  decisions/
  tasks/
  sessions/
  workflows/

README.md는 하네스 지식베이스의 입구다. AI에게도 사람에게도 “어디서 시작할지”를 알려준다.

PROJECT_CONTEXT.md

프로젝트 컨텍스트는 다음 질문에 답한다.

  • 이 프로젝트는 누구를 위해 무엇을 해결하는가?
  • 주요 사용자 흐름은 무엇인가?
  • 시스템 경계는 어디인가?
  • 핵심 도메인 용어는 무엇인가?
  • 바뀌면 위험한 계약은 무엇인가?
  • 실행, 테스트, 배포에서 주의할 점은 무엇인가?
  • 최신 정보와 오래된 정보는 어떻게 구분하는가?

이 문서는 길수록 좋은 문서가 아니다. AI가 매 세션 읽을 수 있어야 하므로 압축되어야 한다.

PROJECT_CONTEXT.md 완성 예시

다음은 실제 팀에서 시작점으로 쓸 수 있는 짧은 예시다. 핵심은 모든 정보를 담는 것이 아니라, 에이전트가 잘못 추측하면 위험한 정보를 먼저 고정하는 것이다.

# PROJECT_CONTEXT

상태: active
최종 검토일: 2026-06-06
소유자: platform-team

## 제품 목적

이 프로젝트는 B2B SaaS 관리자가 조직 멤버, 결제 상태, 접근 권한을 관리하는 웹 애플리케이션이다.
주요 사용자는 고객사의 workspace admin이며, 일반 멤버는 일부 읽기 화면만 접근한다.

## 시스템 경계

- 프런트엔드: React 기반 관리자 콘솔
- 백엔드: REST API
- 인증: 외부 IdP와 사내 세션 API를 함께 사용
- 결제: 별도 billing-service가 소유하므로 이 프로젝트에서 결제 상태를 직접 수정하지 않는다.

## 반드시 지킬 계약

- `workspaceId`는 URL, API 요청, 감사 로그에서 같은 의미로 사용한다.
- 인증 실패 문구는 고객지원 매크로와 연결되어 있어 임의 변경하지 않는다.
- 관리자 권한 변경은 감사 로그에 남아야 한다.

## 자주 깨지는 지점

- 게스트 사용자의 리다이렉트 루프
- 만료 세션과 권한 없음 상태의 혼동
- workspace admin과 organization owner의 권한 혼용

## AI 세션 주의

- 인증, 결제, 권한 변경은 반드시 작업 헌장을 먼저 작성한다.
- `REGRESSION_NOTES.md`의 인증/권한 항목을 확인하지 않은 결과는 통과로 보지 않는다.

이 정도의 문서는 모델에게 “프로젝트의 헌법”처럼 작동한다. 모든 구현 세부사항을 담지는 않지만, 잘못 건드리면 비용이 큰 경계를 선명하게 만든다.

컨텍스트 패킷 만들기

작업 시작 전 다음 형식으로 패킷을 만든다.

이번 작업의 컨텍스트 패킷:

반드시 읽기:
- PROJECT_CONTEXT.md
- tasks/2026-06-auth-email-verification.md
- EVAL_RUBRIC.md

필요하면 읽기:
- decisions/2026-03-auth-policy.md
- REGRESSION_NOTES.md

읽지 말기:
- docs/old-auth-flow.md: 2025년 정책이라 현재와 다름

주의:
- 인증 에러 문구는 고객지원 매크로와 연결되어 있어 임의 변경 금지

이렇게 쓰면 에이전트는 맥락을 덜 추측한다.

컨텍스트 패킷 선택 기준

컨텍스트 패킷은 많이 넣는 게임이 아니다. 다음 기준으로 고른다.

분류포함 기준예시
반드시 읽기완료 조건을 판단하는 데 직접 필요작업 헌장, 평가 루브릭, 관련 ADR
필요하면 읽기구현 선택이나 배경 확인에 도움과거 세션 로그, 유사 PR 설명
읽지 말기오래되었거나 현재 정책과 충돌폐기된 설계 문서, 이전 릴리즈 노트
질문하기문서로 판정할 수 없는 업무 판단정책 변경 승인 여부, 우선순위

가장 흔한 실수는 “관련 있어 보이는 자료”를 전부 넣는 것이다. 관련성보다 판정 가능성이 중요하다. 에이전트가 이번 작업에서 결정을 내리거나 검증할 때 실제로 써야 하는 자료만 우선한다.

지식베이스 운영 규칙

프로젝트 지식베이스는 다음 네 가지 규칙을 가져야 한다.

  • 모든 핵심 문서는 상태, 최종 검토일, 소유자, 적용 범위를 가진다.
  • deprecated 문서는 삭제하지 않더라도, AI가 읽지 않도록 이유를 표시한다.
  • 세션에서 새로 발견한 프로젝트 사실은 SESSION_LOG에 먼저 남기고, 주간 리뷰에서 핵심 문서로 승격한다.
  • 한 문서가 너무 커지면 PROJECT_CONTEXT.md에 요약만 남기고 세부 문서는 decisions/나 workflows/로 분리한다.

이 규칙은 문서가 시간이 지나며 “AI를 돕는 맥락”에서 “AI를 속이는 낡은 맥락”으로 변하는 것을 막는다.

문서 최신성 표시

각 핵심 문서 상단에는 다음 메타 정보를 둔다.

상태: active
최종 검토일: 2026-06-06
소유자: platform-team
적용 범위: checkout, billing, account
주의: legacy checkout v1에는 적용하지 않음

자연어 하네스의 위험 중 하나는 오래된 문서가 강한 권위를 갖는 것이다. 최신성 표시는 작지만 효과적인 방어 장치다.

컨텍스트 부채 관리

컨텍스트 부채는 문서가 많아서 생기는 문제가 아니라, 어떤 문서를 믿어야 하는지 알 수 없어서 생기는 문제다. 팀은 매주 다음 질문을 점검한다.

  • AI가 이번 주에 오래된 문서를 참조했는가?
  • PROJECT_CONTEXT.md에 없는 프로젝트 규칙을 사람이 반복 설명했는가?
  • 세션 로그에 남은 새 사실이 핵심 문서로 승격되지 않았는가?
  • 문서 소유자가 떠났거나 바뀌었는데 소유권이 갱신되지 않았는가?
  • 컨텍스트 패킷에 항상 들어가지만 실제로 쓰이지 않는 문서가 있는가?

이 질문에 답하면 지식베이스는 점점 짧고 강해진다. 좋은 자연어 하네스는 문서를 많이 쌓는 시스템이 아니라, 에이전트가 읽어야 할 맥락을 계속 정제하는 시스템이다.
